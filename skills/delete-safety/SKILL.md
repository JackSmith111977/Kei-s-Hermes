# delete-safety — 删除操作安全边界与确认框架

> **核心原则**：任何删除操作都是危险操作。不同的删除操作有不同的恢复可能性和影响范围。
> AI Agent 必须根据风险等级使用不同的确认机制，不能用一个"确认弹窗"解决所有问题。
>
> **重要警告**：这是 boku 经过深度研究后制定的规则。所有删除操作都必须经过此流程。

## 一、风险分级体系

将删除操作按**可逆性**和**影响范围**分为 4 级：

### S 级：不可逆 + 公开影响 🔴 需双重确认 + 人工审批

| 操作类型 | 示例 | 恢复可能性 |
|----------|------|-----------|
| 删除 GitHub 仓库 / 组织 | `gh repo delete` | ❌ 基本不可恢复 |
| 删除生产数据库 / 存储卷 | `DROP DATABASE` / `volumeDelete` | ❌ 不可恢复 |
| 删除 API 密钥 / 访问令牌 | `gh secret delete` / `kubectl delete secret` | ❌ 影响其他服务 |
| 删除 Release / 版本标签 | `git tag -d` + `git push --delete` | ⚠️ 可恢复但影响大 |
| 危险文件删除操作 | 使用递归删除系统关键文件 | ❌ 不可恢复 |

**确认要求**：
1. 必须**暂停并详细汇报**给用户
2. 汇报必须包含：删除内容、影响范围、是否影响其他服务、恢复方式
3. 要求用户**逐字输入确认词**（如 `YES, DELETE`）而非简单回复"好"
4. 执行后必须报告结果并建议验证

### A 级：不可逆 + 内部影响 🟠 需详细确认

| 操作类型 | 示例 | 恢复可能性 |
|----------|------|-----------|
| 删除有 open PR 的远程分支 | `git push origin --delete feat/active` | ⚠️ 可通过 PR 恢复，但关闭 PR |
| 删除有未合并提交的分支 | `git push origin --delete feat/unmerged` | ⚠️ commits 存在，需 `git reflog` 恢复 |
| 批量删除多个分支 | 一次性删除 2+ 个分支 | ⚠️ 逐一确认 |
| 删除有依赖的配置/Secret | `kubectl delete secret in-use` | ⚠️ 可能导致服务异常 |

**确认要求**：
1. 必须列出**每条**影响：哪些 PR 会被关闭、多少未合并提交
2. 必须说明恢复方式
3. 必须等待用户明确确认
4. 批量删除要求**逐一**确认或用户明确说"全部确认"

### B 级：可逆但有恢复成本 🟡 简短确认

| 操作类型 | 示例 | 恢复可能性 |
|----------|------|-----------|
| 删除已合并的远程分支 | `git push origin --delete feat/merged` | ✅ 可通过 `git reflog` 或 GitHub PR 恢复 |
| 删除本地已合并分支 | `git branch -d feat/merged` | ✅ 可通过 `git reflog` 恢复 |
| 删除本地未推送的废弃分支 | `git branch -D feat/abandoned` | ✅ 可通过 `git reflog` 恢复 |

**确认要求**：
1. 简要汇报分支名 + 状态（已合并/已废弃）
2. 说明可恢复性
3. 等待用户确认

### C 级：安全操作 🟢 无需确认

| 操作类型 | 示例 | 风险 |
|----------|------|------|
| 清理本地远程引用 | `git remote prune origin` | ✅ 无风险 |
| 只读操作 | `git log`, `git branch --merged` | ✅ 无风险 |
| 查询/搜索 | `gh pr list`, `git status` | ✅ 无风险 |

**确认要求**：无需确认，直接执行。

## 二、确认消息模板

### 模板 1：删除已合并分支（B 级）

```
## 🟡 删除确认：{分支名}

### 状态检查
- 关联 PR: #{PR号} — {PR标题}（✅ 已合并到 {目标分支}）
- 未合并提交：0 个
- 最后提交：{日期}

### 可恢复性说明
删除后可通 GitHub PR #{PR号} 页面的 "Restore branch" 按钮恢复，
或通过 `git reflog` 在本地恢复。

### 请确认
是否删除分支 `{分支名}`？
回复 `yes` 确认 / `no` 取消
```

### 模板 2：删除有 open PR 的分支（A 级 — 危险！）

```
## 🟠 危险操作确认：删除分支 `{分支名}`

### ⚠️ 影响范围
- 🔴 该分支有 **{N} 个 open PR**（#{PR号}: "{PR标题}"）
- 🔴 如删除该分支，这些 PR 会被自动关闭
- 📊 该分支有 **{N} 个未合并提交**
- 🔍 最后提交时间：{日期}

### 风险说明
删除后可通过 GitHub PR 页面恢复，
但 PR 的评论 / Review 历史可能会部分影响。

### 请确认
🚀 **是否执行删除操作？**
- 输入 `yes` — 确认删除（会关闭 {N} 个 PR）
- 输入 `no` — 取消操作
- 输入 `wait` — 保留稍后处理
```

### 模板 3：批量删除（多个分支）

```
## ⚠️ 批量删除确认：{N} 个分支将被删除

### 清单
| # | 分支名 | 状态 | 关联 PR | 风险等级 |
|---|--------|------|---------|---------|
| 1 | `{分支1}` | ✅ 已合并 | #{PR号} | 🟡 B 级 |
| 2 | `{分支2}` | ✅ 已合并 | #{PR号} | 🟡 B 级 |
| 3 | `{分支3}` | ❌ 已放弃（无 PR） | 无 | 🟡 B 级 |

### 可恢复性说明
所有被删分支的 commits 仍存在于 GitHub 上，可通过 PR 页面或 reflog 恢复。

### 请逐一确认
请在检查后逐条回复确认：
1. `确认1` — 删除 {分支1}
2. `确认2` — 删除 {分支2}
3. `确认3` — 删除 {分支3}
```

### 模板 4：删除仓库 / 存储卷（S 级 — 极度危险！）

```
## 🔴🔴 极度危险操作确认 🔴🔴

### 要删除的内容
- 类型: GitHub 仓库 / 数据库 / 存储卷
- 名称: {名称}
- 影响范围: **所有代码/数据将被永久删除**

### 风险等级: S 级 — 不可逆 + 公开影响

⚠️ 此操作**无法撤销**！
⚠️ 删除后可能影响所有团队成员和下游服务

### 请确认
请输入 **大写** `YES DELETE` 确认操作：
（输入任何其他内容将取消操作）
```

## 三、AI Agent 操作铁律

### 3.1 黄金规则（必须遵守）

```
1. ✅ 所有删除操作必须暂停并汇报（C 级除外）
2. ✅ 汇报必须包含：删除内容、影响范围、恢复可能性、替代方案
3. ✅ 必须等待用户明确确认后才能执行
4. ✅ 执行后必须报告结果
5. ✅ 批量删除必须逐一列明并获取确认
```

### 3.2 确认消息必须包含的 6 要素

| 要素 | 说明 | 必须 |
|------|------|------|
| **WHO** | 谁提出的删除 | ✅ |
| **WHAT** | 要删除什么 | ✅ |
| **WHY** | 为什么要删除 | ✅ |
| **IMPACT** | 影响范围 | ✅ |
| **RECOVER** | 能否恢复 + 如何恢复 | ✅ |
| **CONFIRM** | 用户如何确认 | ✅ |

### 3.3 绝对禁止的行为 🚫

- ❌ **用户没说"好"就执行删除**
- ❌ 用模糊的确认词替代明确确认（但主人说"嗯"或"好"时，boku 必须理解为已确认——**但必须以详细的确认消息呈现选项**）
- ❌ 批量删除时**不逐一列明**影响
- ❌ 用户说"都删了"时**不核对数量**（必须先列出清单让主人过目）
- ❌ **不报告删除结果**（要列出哪些成功了、哪些失败了）
- ❌ 删除后**不说明恢复方式**

### 3.4 确认疲劳防御

根据"工具分类学"原则，不同风险级别使用**不同形式的确认**：

| 级别 | 确认形式 | 说明 |
|------|----------|------|
| 🟢 C 级 | 无需确认 | 直接执行 |
| 🟡 B 级 | 详细列表 + 一次确认 | "请确认删除以上分支" |
| 🟠 A 级 | 详细影响分析 + 逐一确认 | 每个分支单独询问 |
| 🔴 S 级 | 多重确认 + 输入特定文字 | 必须输入 `YES DELETE` |

不要对所有删除操作使用同一个"确认弹窗"——这会导致主人对确认信息产生肌肉记忆，不再仔细查看。

## 四、Git 分支删除的安全操作指南

### 4.1 安全检查步骤

在执行任何分支删除前，必须完成以下检查：

```bash
# 1. 检查分支的合并状态
git branch --merged master | grep <branch-name>
git branch --no-merged master | grep <branch-name>

# 2. 检查是否有 open PR（通过 GitHub API）
gh pr list --head <branch-name> --state open --json number,title

# 3. 检查远程是否存在
git ls-remote --heads origin <branch-name>

# 4. 获取最后提交信息
git log origin/<branch-name> -1 --oneline 2>/dev/null || echo "分支不存在或已删除"
```

### 4.2 删除方式对比

| 方式 | 命令 | 影响 | 安全建议 |
|------|------|------|----------|
| 本地安全删除 | `git branch -d <name>` | 仅本地，有安全检查 | ✅ 推荐 |
| 本地强制删除 | `git branch -D <name>` | 仅本地，跳过检查 | ⚠️ 谨慎使用 |
| 远程分支删除 | `git push origin --delete <name>` | 影响远程仓库 | ⚠️ 需确认 |
| GitHub API 删除 | `DELETE /git/refs/heads/<name>` | 影响远程仓库 | ⚠️ 需确认 |
| GitHub UI 删除 | PR 页面的 Delete branch 按钮 | 影响远程仓库 | ✅ 安全（有确认） |

### 4.3 恢复方式速查

| 恢复方式 | 适用场景 | 命令 |
|----------|----------|------|
| GitHub PR 恢复 | 有关联 PR 的分支 | PR 页面 → "Restore branch" 按钮 |
| `git reflog` 恢复 | 本地有缓存的分支 | `git reflog` → 找 hash → `git checkout -b <name> <hash>` |
| `git log --all` 恢复 | 其他分支有引用 | `git log --all --decorate --oneline --graph` → 找 hash |
| `git fsck` 恢复 | 未 GC 的对象 | `git fsck --full` → 找 dangling commit |

## 五、实战检查清单

每次删除前，先回答以下问题：

- [ ] 删除的是什么？（分支 / 文件 / 资源 / 仓库）
- [ ] 这个操作可逆吗？
- [ ] 影响范围多大？（仅本地 / 仅团队成员 / 影响外部用户）
- [ ] 有没有关联的 PR / Issues / 其他资源？
- [ ] 有没有替代方案？
- [ ] 用户确认了吗？
- [ ] 执行后需要通知用户结果吗？
