# Cron 反射门禁自检模式

> **问题**：夜间自习引擎 v3.x 依赖 `learning-workflow/scripts/reflection-gate.py` 执行 R1/R2/R3/L3 自动门禁评分。但该脚本要求**活跃学习任务**（由 `learning-state.py init` 创建）。Cron 环境中通常不初始化学习任务，导致 `reflection-gate.py r1` 返回 `{"error": "无活跃学习任务"}`。
>
> **解决方案**：使用结构化自检替代脚本自动化门禁。以 R1/R2/R3/L3 的原始检查标准为基准，通过代查清单完成质量验证。

## 触发条件

```bash
python3 ~/.hermes/skills/learning-workflow/scripts/reflection-gate.py r1 <task_id>
# → {"error": "无活跃学习任务"}
```

或当不想（或不能）初始化学习状态机时。

## 自检流程

### R1 自检 — 搜索质量（搜索后）

| 检查项 | 通过标准 | 自检方法 |
|:-------|:---------|:---------|
| 数量 | ≥ 3 个来源 | 统计 `web_search` 返回结果数或 extract 的独立来源数 |
| 权威性 | ≥ 1 个官方来源 | 检查 URL 是否来自 `openai.com`、`aws.amazon.com`、`cloud.google.com`、`blogs.nvidia.com` 等官方域名 |
| 覆盖度 | 覆盖主题各方面 | 对照 knowledge_map.md 的子主题列表检查是否都有覆盖 |
| 时效性 | < 12 个月 | 检查发布日期（搜索摘要中通常包含日期） |

```text
R1 自检模板：
✅ 数量: X 个来源 (≥3? pass)
✅ 权威性: X 个官方来源 (≥1? pass)
✅ 覆盖度: 覆盖了 N/M 个子主题
✅ 时效性: 全部来自最近 X 天/月内 (<12m? pass)
→ R1 判定: ✅ 通过 / 🔴 需要回退
```

### R2 自检 — 理解深度（阅读后）

| 检查项 | 通过标准 | 自检方法 |
|:-------|:---------|:---------|
| 核心概念 | 能用自己的话解释 | `reading_notes.md` 中有清晰的概念定义而非原文复制 |
| 可操作性 | 能写出操作步骤 | 包含选型建议/对比表/步骤描述（研究型会话可加"场景建议"） |
| 完整性 | 无明显知识缺口 | 对照 knowledge_map.md 子主题检查是否全部覆盖 |
| 交叉验证 | ≥ 2 个来源佐证 | 对关键结论标注多个来源 URL |

```text
R2 自检模板：
✅ 核心概念: reading_notes 有自己的归纳（而非复制粘贴）?
✅ 可操作性: 有选型表/步骤/场景建议?
✅ 完整性: 覆盖所有子主题?
✅ 交叉验证: 关键结论有 ≥ 2 来源?
→ R2 判定: ✅ 通过 / 🔴 需要回退
```

### R3 自检 — 提炼完整性（提炼后）

extracted_knowledge.md 必须有以下五大章节（缺失任何一章即不通过）：

```markdown
## 一、🆕 核心新增概念（含来源标记🥇🥈🥉）
## 二、🔄 到期概念升级建议（含交叉验证引用）
## 三、💡 应用场景与选型建议（可操作）
## 四、⚠️ 待观察与风险提示
## 五、📊 质量评分（四维度自评）
```

| 检查项 | 通过标准 |
|:-------|:---------|
| 结构 | 5 章齐全（含#六执行清单为加分项） |
| 可操作性 | 第三/四章有具体建议或风险提示 |
| 完整性 | 无明显的概念遗漏或知识缺口 |

### L3 质量门禁自检 — 递进评分

当 reflection-gate.py 不可用时，使用四维度评分公式：

```
质量分 = 信息覆盖度(30) + 交叉验证(25) + 可操作性(25) + 结构完整度(20)
```

| 维度 | 评分指南 | 满分 |
|:-----|:---------|:---:|
| 信息覆盖度 | 覆盖主题各子方向（2-3 个=22, 4-5 个=27, 6+ 个=30） | 30 |
| 交叉验证 | 关键结论有 ≥2 来源（部分=22, 全部=25） | 25 |
| 可操作性 | 有选型表/步骤/场景建议（一般=20, 详细=25） | 25 |
| 结构完整度 | 5 章齐全且质量高（缺1章=15, 全齐但简=18, 全齐详细=20） | 20 |

**递进规则**（手动执行）：
| 循环 | 通过门槛 | 策略 |
|:---:|:--------:|:-----|
| 第 1 次 | raw ≥ 60 | 强制通过，标记"进入第 2 轮" |
| 第 2 次 | raw ≥ 70 | 标准严格 10 分 |
| 第 3 次 | raw ≥ 80 | 标准严格 20 分 |

**⚠️ Cron 环境限制**：L3 递进循环需要第 2 轮重新搜索+阅读+提炼，这在单次 cron 执行中通常不现实。建议：
- 第 1 轮 self-check（raw ≥ 60）→ 产出 Artifact + 写入 session_log → 完成
- 如果 raw < 60 → 更换搜索关键词，重试 1 次（Loop N+1）
- 仍然 < 60 → 写入 gap_queue，下次 crontab 轮次继续

## 与 learning-workflow 反射门禁的关系

| 方面 | reflection-gate.py 自动门禁 | 手动自检模式 |
|:-----|:--------------------------|:------------|
| 评分方式 | 脚本自动解析文件内容 | 手动对照清单打分 |
| 子代理仲裁 | 40-70 分时自动启动 | 不可用——cron 环境无 delegate_task |
| 递进循环 | 自动追踪 loop_count | 手动记录 |
| 推荐场景 | 交互式学习任务 | Cron 自学任务（无 learning-state） |

**不建议的场景**：交互式学习任务（用户在线时）仍应使用 `learning-state.py init` + `reflection-gate.py` 标准流程。自检模式是 cron 环境下的降级方案。

## 与 R2 可操作性假阴性的关系

本参考与 `research-session-patterns.md` 的 R2 绕过策略互补：
- **research-session-patterns.md** 解决 R2 对研究型会话的"可操作性"假阴性问题（无代码示例时得分低）
- **本参考** 解决 reflection-gate.py 在 cron 环境中完全不可用的问题

叠加使用时：先按本参考做 R2 自检，若可操作性维度因无代码示例得分低，再应用 research-session-patterns.md 的缓解策略。
