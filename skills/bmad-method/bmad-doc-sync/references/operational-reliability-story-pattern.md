# 运维观察驱动的可靠性 Story 编写模式

> 当通过**运行时观察**（而非 PRD/需求文档）发现可靠性问题时，如何将其结构化成 BMAD Story 并追加到现有 Epic。

## 适用场景

- 发现进程重复启动（如 3 个 SRA 实例同时运行）
- 发现竞态条件（PID 检查无原子锁）
- 发现异常退出后的资源泄漏（锁文件未释放）
- 发现端口/文件冲突（多实例争抢同一端口）
- 任何「观察现象 → 追溯代码 → 确认漏洞」的逆向发现流程

## 与标准 BMAD Story 编写的区别

| 维度 | 需求驱动的 Story | 运维观察驱动的 Story |
|:---|:---|:---|
| 来源 | PRD / 用户需求 | `ps aux` / 日志 / 异常行为 |
| 分析方向 | 正向：需求 → 方案 | 逆向：现象 → 根因 |
| 验收标准 | 功能行为描述 | 防御性约束描述 |
| 优先级 | 按业务价值定 | 按风险等级定（观察到的实际故障 = P0） |
| 估时 | 功能复杂度 | 往往 0.5d 左右（修复比重构小） |

## Story 编写模板

### 结构

```markdown
### Story N: [标题] — [子标题] (SRA-003-0N)

> **作为** [受影响的主体]
> **我希望** [期望的行为]
> **以便** [解决的问题]

**问题背景：** [观察到的现象 + 当前实现的具体漏洞描述]

**根因分析：**

| # | 漏洞 | 场景 | 严重程度 |
|:---|:---|:---|:---:|
| 1 | [漏洞名称] | [什么情况下触发] | 🔴 竞态/多实例/崩溃 |
| 2 | [第二个漏洞] | [触发场景] | 🟡 无声失败 |

**验收标准：**
- [ ] **核心修复**: 具体的技术方案 + 实现方式
- [ ] 第二个验收项
- [ ] 第三项
- [ ] **优雅降级**: 检测到异常时新进程的行为（退出码、提示信息）
- [ ] **测试用例**: 需要覆盖的并发/边界场景

**实现文件：**
- 修改: `path/to/module.py`（改动说明）
- 新增: `path/to/new_module.py`（新模块说明）
```

### 关键原则

1. **现象第一** — 开头就说观察到的具体现象（"3 个 SRA 实例同时运行"），不要说抽象概念
2. **根因表格** — 用表格列出每个漏洞的触发场景和严重程度，比段落描述清晰 10 倍
3. **防御性验收标准** — 这类 Story 的验收标准不是「功能实现」，而是「防止 xxx 情况发生」
4. **优雅降级必写** — 任何失败都要有可读的退出信息，不能静默崩溃
5. **优先级定 P0** — 已经实际发生的问题就是 P0，不管它看起来多「小」

## 在已有 Epic 中追加的流程

```bash
1. 观察现象          # ps aux | grep sra → 发现 3 个进程
2. 追溯代码          # 读 daemon.py 的 cmd_start/cmd_attach
3. 列出漏洞          # 根因分析表格
4. 编写 Story        # 按上述模板
5. 插入 Epic 文档    # patch EPIC-00N.md 的故事列表
6. 更新验收总表      # 添加 Story ID、优先级、估时
7. 更新 ROADMAP      # 如果当前有 Sprint 表
8. 更新发布计划      # Sprint 分割表
```

## 实战案例：SRA-003-12 (Daemon 单例守护)

完整实现在 `~/projects/sra/docs/EPIC-003-v2-enforcement-layer.md` 的 Story 12 段。

根因分析表：

```
| # | 漏洞               | 场景                         | 严重程度                 |
| 1 | PID 检查无原子锁    | sra start 两次快速调用       | 🔴 竞态 → 多实例        |
| 2 | cmd_attach 无检查   | systemd sra attach 调用      | 🔴 无条件多重启动       |
| 3 | 端口级无保护        | HTTP 端口被残骸占用          | 🟡 启动失败无声         |
| 4 | _update_status 无归属 | 多个实例同时写 status.json | 🟡 状态文件损坏         |
```

核心修复方案：`fcntl.flock` OS 级文件锁 + 端口活性探测 + `atexit` 自动释放。
