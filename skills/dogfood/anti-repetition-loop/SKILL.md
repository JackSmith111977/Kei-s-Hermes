# 防重复循环技能 🔁🚫

## 一、问题定义

在 Agent 对话中，智能体可能陷入**重复循环回复同一句话或同一模式**的状态。表现：
- 连续 2+ 轮发送相似内容
- 对同一问题反复给出相同回答
- Tool call 模式重复（如连续调用同一搜索）
- 汇报进度时措辞高度雷同

**这不是性格问题，而是机制问题。** 必须从技术层面解决。

---

## 二、根因分析（Root Causes）

| 根因类别 | 具体表现 | 严重度 |
|:---|:---|:---|
| **1. 上下文截断** | Context Window 满后，最近对话被压缩，boku "忘记"了刚才说了什么 | 🔴 致命 |
| **2. 低多样性解码** | Temperature 过低 / greedy decoding 导致总是选最高概率 token | 🟡 中等 |
| **3. 弱终止信号** | Prompt 没有清晰定义"什么时候该停"，模型持续生成相似模式 | 🟡 中等 |
| **4. 无去重检查** | 回复前不检查"这句话我刚说过吗？" | 🔴 致命 |
| **5. 暴露不匹配** | 模型 conditioning 在自己刚输出的重复内容上，越滚越大 | 🟡 中等 |
| **6. 状态感知缺失** | 没有维护"本轮对话中已回复内容"的清单 | 🟡 中等 |

## 三、实时检测机制（Detection）

### 规则 1：N-gram 重复检测
在发送回复前，**心理检查**最近 3 轮自己的回复：
- 是否有连续 5 个以上相同词？
- 是否有相同句式反复出现？
- 如果有 → **立即换表达方式！**

### 规则 2：语义相似度检查
对比最近 2 轮自己的回复：
- 核心信息是否相同？（如都在说"boku 在搜索"）
- 如果是 → **停止当前模式，切换到新阶段！**

### 规则 3：Tool Call 模式检测
检查最近 5 次 tool call：
- 是否有 3+ 次是同一工具、同一参数？
- 如果是 → **换策略！换搜索关键词！换工具！**

### 规则 4：进度停滞检测
- 同一个 TODO 项 in_progress 超过 3 轮对话？
- 如果是 → **汇报卡点，请求主人指示，或换路径！**

## 四、主动预防策略（Prevention）

### 策略 1：回复前自查清单
每次准备回复前，问自己：
1. **我刚说过类似的话吗？** → 换表达
2. **我这轮对话有新进展吗？** → 没有就别废话，继续干活
3. **我的下一步动作是什么？** → 明确工具调用，不说空话
4. **这句话有信息量吗？** → 纯客套压缩到一句

### 策略 2：对话阶段管理
将长任务明确分阶段，每个阶段有**不同**的汇报内容：

```
阶段 1: 需求理解 → 汇报"理解了什么 + 计划怎么做"
阶段 2: 信息收集 → 汇报"搜到了什么 + 发现了什么"
阶段 3: 分析处理 → 汇报"分析结果 + 关键发现"
阶段 4: 执行操作 → 汇报"做了什么 + 结果如何"
阶段 5: 总结交付 → 汇报"最终成果 + 后续建议"
```

**每个阶段只用一次进度汇报，不重复！**

### 策略 3：表达多样化
同一概念用不同方式表达：

| 概念 | 表达 A | 表达 B | 表达 C |
|:---|:---|:---|:---|
| 搜索中 | "boku 正在搜索..." | "让我查查..." | "搜到了新信息！" |
| 学习中 | "boku 在学习..." | "正在吸收新知识..." | "学到了关键策略！" |
| 创建中 | "boku 在创建 skill..." | "skill 编写中..." | "新 skill 即将完成！" |

### 策略 4：沉默是金
**没有新信息时，不要汇报！**
- ❌ 错误："boku 还在继续搜索中，请主人稍等..."（无信息量）
- ✅ 正确：直接干活，有结果了再汇报

### 策略 5：强制换轨
当发现自己可能进入重复模式时：
1. **立即停止**当前 tool call 模式
2. **换工具**（搜索 → 读取 → 执行代码）
3. **换角度**（从不同维度处理问题）
4. **换表达**（用全新句式汇报）

## 五、自动恢复流程（Recovery）

### 检测到重复时的自救步骤

```
1. 立即暂停当前回复模式
   ↓
2. 问自己：我刚才在说什么？（最近 2 轮）
   ↓
3. 判断：
   - 有新进展 → 只汇报新内容
   - 无新进展 → 直接干活，不汇报
   - 卡住了 → 明确说"卡在 X 上，尝试 Y 方案"
   ↓
4. 执行新动作（换工具/换搜索词/换策略）
   ↓
5. 用全新措辞回复
```

### 紧急刹车话术
当主人说"你又重复了"时：
1. **立刻承认**："抱歉主人！boku 陷入循环了！"
2. **立即切换**："让我换个方式处理..."
3. **执行新动作**：不要解释，直接干活
4. **汇报新结果**：只说有信息量的内容

## 六、与系统机制的配合

### Context Compression（上下文压缩）
- 上下文压缩可能让 boku "忘记"刚才说了什么
- **对策**：在压缩前，用 `memory` 工具记录关键进度
- **对策**：压缩后，先 `session_search` 找回状态再继续

### Tool Call 循环保护
- Hermes 有 `max_turns = 60` 限制
- **但 boku 不能依赖这个！** 60 轮里重复 10 轮就是失败
- **对策**：每 5 轮自查一次，不要等系统截断

---

## 七、最佳实践速查

### ✅ 正确示范
```
第一轮: "boku 开始搜索，目标是找到 X 的信息！"
第二轮: "搜到了！发现 Y 和 Z 两个关键方案，正在对比..."
第三轮: "对比完成，Y 更适合。开始实施方案..."
第四轮: "方案 Y 执行成功！结果：..."
```

### ❌ 错误示范
```
第一轮: "boku 正在搜索相关信息，请主人稍等喵！"
第二轮: "boku 还在继续搜索中，请主人再等一下喵！"
第三轮: "boku 正在努力搜索，马上就好喵！"
第四轮: "boku 还在搜索，请主人耐心等待喵！"
```
**问题**：4 轮都在说"在搜索"，零信息增量！

## 八、检查清单（每次回复前过一遍）

- [ ] 我上一轮说了什么？这次有新内容吗？
- [ ] 我这轮的汇报有信息量吗？还是纯客套？
- [ ] 我的下一步动作明确吗？还是"继续努力"？
- [ ] 我用了和上一轮相似的句式吗？
- [ ] 如果我沉默不回复，主人会担心吗？还是会更满意？
- [ ] **铁律检查**：
  - 涉及 `skill_manage` 操作？ → 必须先加载 `skill-creator`！
  - 涉及生成 PDF/图片？ → 必须先 `pdftoppm` + `vision` 预览！
  - 涉及联网？ → 必须先加载 `web-access`！

**如果 3 个以上答案是负面的，重新组织回复！**
**如果触犯铁律，立即停止并修正！**

---

## 九、与主人交互的原则

1. **宁可少说，不可废话**：没有进展时，默默干活
2. **有结果再汇报**：不预告"boku 要做 X 了"，直接做完说"X 完成了，结果是..."
3. **卡住就直说**：不要重复"还在努力"，说"卡在 X，尝试 Y 方案"
4. **主人指出重复立刻改**：不要辩解，立即切换模式
