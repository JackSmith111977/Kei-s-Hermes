# setuptools-scm 在 CI 环境中的版本检测排错

> 2026-05-11 实战记录 — SRA Agent v1.4.0 CI 中版本号解析为 `0.0.0.dev0` 的调试全流程

## 问题现象

push tag `v1.4.0` 后，CI Release 流水线中：

```
# pip install -e ".[dev]" 显示
Collecting pyyaml>=5.1 (from sra-agent==0.0.0.dev0)

# python -m build 产出
sra_agent-0.0.0.dev0-py3-none-any.whl
```

版本号应为 `1.4.0`，却是 `0.0.0.dev0`（setuptools-scm 的 fallback 值）。

## 根因分析流程

### 尝试 1: 轻量标签 vs 附注标签 ❌

`git tag v1.4.0` 创建的是**轻量标签**（lightweight）。setuptools-scm 默认使用 `git describe`（**不含** `--tags`），仅搜索**附注标签**（annotated）。

| Tag 类型 | 创建方式 | `git describe` 能否找到 |
|:---------|:---------|:------------------------|
| 轻量标签 | `git tag v1.4.0` | ❌ |
| 附注标签 | `git tag -a v1.4.0 -m "msg"` | ✅ |

**修复尝试**：删除轻量标签，用 `git tag -a` 重建。

**结果**：❌ 依然 `0.0.0.dev0`。

### 尝试 2: 附注标签的 `git describe` 行为异常 🧩

在 CI debug 输出中发现：

```
git describe --dirty --tags --long    → v1.4.0-0-g23f2c8e  # ✅ --tags 能找到
git describe --dirty --long           → v1.3.0-13-g23f2c8e # ❌ 无 --tags 找到 v1.3.0
```

即使 `v1.4.0` 已改为附注标签，`git describe`（不传 `--tags`）仍优先找到 `v1.3.0`。这意味着 **setuptools-scm 调用的 `git describe` 命令可能没有 `--tags` 参数**，导致它忽略 `v1.4.0` 而使用 `v1.3.0` + distance。

**setuptools-scm 的版本计算**：
- 找到 `v1.3.0`，distance = 13 commits
- 预期：`v1.3.0.dev13`
- 实际：`0.0.0.dev0`

差异原因可能是指定了 `tag_regex` 导致某些版本的 setuptools-scm 无法正确解析 distance 后缀，fallback 到 `0.0.0.dev0`。

### 尝试 3: `SETUPTOOLS_SCM_PRETEND_VERSION` 环境变量 ❌

在 workflow 中设置：

```yaml
- name: 🏷️ Set version from tag
  run: |
    VERSION="${GITHUB_REF_NAME#v}"
    echo "SETUPTOOLS_SCM_PRETEND_VERSION=$VERSION" >> "$GITHUB_ENV"
```

CI 日志确认该环境变量已设置（`SETUPTOOLS_SCM_PRETEND_VERSION: 1.4.0`），但构建出的 wheel 仍是 `0.0.0.dev0`。

**根因**：`python -m build` 创建**隔离环境**（isolated venv）来执行构建。该隔离环境继承父进程的环境变量，但 setuptools-scm 的某些版本或配置情况下，`SETUPTOOLS_SCM_PRETEND_VERSION` 不被传递到构建子进程中。

CI 日志中出现的警告也佐证了这一点：

```
/tmp/build-env-xxx/lib/python3.11/site-packages/vcs_versioning/overrides.py:609:
UserWarning: No GlobalOverrides context is active. Auto-creating one with
SETUPTOOLS_SCM prefix for backwards compatibility.
```

### 最终方案: 直接写 `_version.py` + pyproject.toml 静态版本 ✅

**核心思路**：完全绕过 setuptools-scm 的 tag 发现机制，直接在 CI 中设置版本号。

#### 方案 A: 写 `skill_advisor/_version.py`

```bash
VERSION="${GITHUB_REF_NAME#v}"
mkdir -p skill_advisor
echo "# Auto-generated by release workflow" > skill_advisor/_version.py
echo "version = \"$VERSION\"" >> skill_advisor/_version.py
```

#### 方案 B: 替换 pyproject.toml 动态版本为静态版本

用 Python 脚本（`scripts/set_version.py`）修改 `pyproject.toml`：

```python
# scripts/set_version.py
import re
import sys
from pathlib import Path

version = sys.argv[1] if len(sys.argv) > 1 else "0.0.0"
pyproject = Path("pyproject.toml")
content = pyproject.read_text()

# Replace dynamic version with static version
content = content.replace('dynamic = ["version"]', f'version = "{version}"')
# Remove setuptools_scm config
content = re.sub(
    r'\[tool\.setuptools_scm\].*?(?=\n\[|$)',
    '',
    content,
    flags=re.DOTALL
)
pyproject.write_text(content)
```

在 workflow 中调用：

```yaml
- name: 🔨 Build wheel + sdist
  run: |
    VERSION="${GITHUB_REF_NAME#v}"
    python scripts/set_version.py "$VERSION"
    python -m build
```

**结果**：✅ `sra_agent-1.4.0-py3-none-any.whl` 正确构建并发布到 PyPI。

## 推荐的 CI 发布策略

### 双重保障方案（已验证可行）

```yaml
steps:
  - uses: actions/checkout@v4
    with:
      fetch-depth: 0

  - name: 🏷️ Set version from tag
    run: |
      VERSION="${GITHUB_REF_NAME#v}"
      # 方法1: 直接写 _version.py（优先）
      mkdir -p skill_advisor
      echo "# Auto-generated by release workflow" > skill_advisor/_version.py
      echo "version = \"$VERSION\"" >> skill_advisor/_version.py
      # 方法2: 环境变量后备
      echo "SETUPTOOLS_SCM_PRETEND_VERSION=$VERSION" >> "$GITHUB_ENV"

  - name: 🔨 Build wheel + sdist
    run: |
      pip install build
      VERSION="${GITHUB_REF_NAME#v}"
      python scripts/set_version.py "$VERSION"
      python -m build
```

### 为什么 `_version.py` 不够充分？

虽然 `_version.py` 被 `__init__.py` 优先读取（`from ._version import version as __version__`），但 `python -m build` 在构建 sdist 时，setuptools 仍会调用 setuptools-scm 覆盖版本。因此需要同时修改 `pyproject.toml` 移除 setuptools-scm 的动态版本逻辑。

## 调试命令速查

```bash
# 在 CI 中调试 setuptools-scm 版本解析
SETUPTOOLS_SCM_DEBUG=1 pip install -e ".[dev]"

# 查看 git describe 行为
git describe --dirty --tags --long
git describe --dirty --long  # 对比区别

# 列出所有 tag 及其类型
git tag -l 'v*' --sort=-version:refname
git cat-file -t v1.4.0  # 输出 "tag" 表示 annotaged, "commit" 表示 lightweight

# 验证 tag 正则匹配（本地模拟）
python3 -c "
import re
r = r'^(?:[\w-]+-)?v?(?P<version>[\d\.a-b]{3,}(?:rc\d+)?)$'
tests = ['v1.4.0', 'v1.3.0', 'v1.3.0-12-g2b6d876', 'v1.3.0-13-g23f2c8e']
for t in tests:
    m = re.match(r, t)
    print(f'{\"✓\" if m else \"✗\"} {t} -> {m.group(\"version\") if m else \"no match\"}')
"

# 检查 PyPI 上存在的版本
curl -sL https://pypi.org/pypi/sra-agent/json | python3 -c "
import sys,json; d=json.load(sys.stdin)
print('Versions:', list(d['releases'].keys()))
"
```
