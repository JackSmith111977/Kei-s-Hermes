# Detailed Reference

## 阶段 2：诊断工具箱 🧰

不同的问题类型用不同的诊断方法：

### 类型 A：工具不好用 / 不可靠

```
症状: 某个工具总是出问题
诊断方法:
  1. 搜索"工具名 + 替代方案"（web-access → Tavily Search）
  2. 对比 2-3 个替代方案的优缺点
  3. 验证最合适的方案是否可用（terminal 测试）
  4. 创建/更新 skill 记录最佳实践
```

**实战案例：** patch 工具污染文件 → 搜索替代方案 → 发现 write_file 更安全 → 创建 `patch-file-safety` skill

### 类型 B：搜索结果不对 / 找不到答案

```
症状: 搜了几次都找不到想要的信息
诊断方法:
  1. 换搜索词：用更具体的术语、加引号精确匹配
  2. 换搜索源：官方文档 > 权威教程 > GitHub Issues > Stack Overflow
  3. 换搜索方式：Tavily Search → Tavily Extract（精读）→ Tavily Crawl（系统爬取）
  4. 分治法：把大问题拆成 N 个小问题分别搜
```

**实战案例：** 搜 Python 最佳实践 308KB 结果 → 分 pathlib/logging/type hints/error handling 四个小主题分批搜

### 类型 C：方案不确定 / 不知道对不对

```
症状: 有多个方案可选，不知道哪个最好
诊断方法:
  1. 列出所有方案的优缺点（表格对比）
  2. 每个方案做最小可行性测试（最快的那个先试）
  3. 交叉验证：找 2-3 个不同来源确认
  4. 问主人（clarify）——boku 觉得不擅长的决策不要硬猜
```

**实战案例：** PDF 用 ReportLab 还是 WeasyPrint？→ 对比表格 → WeasyPrint 胜出 → 创建 `pdf-layout-weasyprint` skill

### 类型 D：卡住了 / 完全没思路

```
症状: 不知道从哪里下手
诊断方法:
  1. 回到"最小可工作版本"——删掉所有无关东西
  2. 找类似问题的已有案例（session_search）
  3. 分解：把大问题切成小步骤，从第一步开始
  4. 用 delegate_task 让子代理从不同角度探索
  5. 跟主人说"boku 卡住了，需要你指点一下"
```

**实战案例：** PDF 中文黑块 → 删掉所有样式回到纯文字 → 发现是字体问题 → 定位到 Helvetica

### 类型 E：反复失败 / 改了好几次都不行

```
症状: 同一个问题改了 3 次还没好
诊断方法:
  1. STOP! 第 4 次修复之前必须回溯
  2. 检查假设——之前的"根因"判断可能错了
  3. 加日志/打印——看中间状态
  4. 用 systematic-debugging 的 Phase 4 Step 5：
     连续 3+ 次修复失败 → 可能是架构问题，不是 bug
```

**实战案例：** fallback 不触发 429 → 试了 3 次 patch 都不对 → 发现 `key_env` 没有被解析 → 应该直接 patch 源码而非改 config

### 类型 F：系统设计问题 — 文档层 vs 代码层落差

```
症状: 系统文档里描述得很完美，但运行时就是达不到预期效果
诊断方法 — 5 层剥洋葱法:
  1. 🎭 症状层：用户看到什么表面现象？（循环没跑起来）
  2. 📄 文档层：SKILL.md/文档承诺了什么？（R1/R2/R3/质量门禁）
  3. 📂 代码层：实际代码实现了什么？（learning-state.py 只有线性推进）
  4. 📋 状态机层：数据结构支持什么操作？（只有 pending/completed，没有 needs_revision）
  5. 🧠 心理层：执行者的潜意识驱动力是什么？（"完成主义"——想尽快走完流程）
```

**核心洞察**：系统不按预期运转常常不是因为文档写得不好，而是因为**文档承诺了什么和代码实现了什么之间存在鸿沟**。修复顺序要倒过来：先让代码能表达"循环"（状态机 API），再让文档描述正确的循环。

**特别诊断技巧 — 追踪数据流**：
```python
# 当文档说"如果 R1 失败就回到 STEP 0"
# 但状态机里没有 regress 命令时，
# 沿着数据流看每一步：
文档 → 解释了应该做什么
代码 → 只实现了 complete → pending
结论 → 循环只在文档中存在，代码里不存在
```

**实战案例**：learning-workflow v4.0 循环不运转 → 5 层分析发现文档层有 R1/R2/R3/质量门禁，但代码层只有线性推进 → v5.0 修复：加状态机 API + 反射门禁脚本 + 子代理裁判

### 类型 G：自评自判陷阱 (Self-judging Bias)

```
症状: 质量检查永远都是"通过"，从不触发回退
根因: 同一实体既做执行者又做裁判
诊断方法:
  1. 检查检查者：谁在做质量评估？是和执行者同一个人吗？
  2. 检查标准：评估标准是明确可量化的（代码评分），还是模糊主观的（"我觉得够了"）？
  3. 检查后果：评估失败会带来什么后果？会不会增加执行者的工作量？
  
  如果三者都是"自己人""模糊""增加工作量" → 100% 自评自判陷阱
```

**为什么自评自判必然失败**：
```python
# 潜意识计算：
通过 = 进入下一步 ✓ （省事、进度推进）
不通过 = 回到上一步 ✗ （更多工作、进度倒退）

# 当"是否通过"由自己判断时...
# → 总是选择"通过"，因为这是认知负担最小的选项
```

**打破自评自判的三种方式**：

| 方式 | 实现 | 成本 | 效果 |
|:----:|:----|:----:|:----:|
| **自动化脚本** | 硬编码的客观评分规则（reflection-gate.py） | 低 | ⭐⭐⭐ |
| **子代理裁判** | 用独立的 Agent 做第三方评估 | 中 | ⭐⭐⭐⭐ |
| **人类审核** | 最终由人类决定 | 高 | ⭐⭐⭐⭐⭐ |

**实战案例**：learning-workflow v4.0 的 R1/R2/R3 检查点 → boku 自评自判 → 永远通过 → v5.0 引入 reflection-gate.py + 子代理裁判 → 真正触发了质量门禁

## 阶段 3：解决方案生成 💡

### 方案来源优先级

```
1. 已有 skill / memory（最可靠）
2. 官方文档（最权威）
3. 之前成功过的经验（session_search）
4. 类似问题的在线解法（Tavily Search）
5. 尝试性方案（最小测试）
6. 问主人（最后选择）
```

### 最小验证原则

```python
# 不要一上来就写完整方案
# 先做最小验证

# ❌ 写 100 行脚本 → 出 bug → 排查 30 分钟
# ✅ 写 5 行测试 → 确认方向对 → 再写完整实现
```

## 阶段 4：学习与沉淀 📝

**每次解决问题后，必须问自己：**

1. **根因是什么？**（一句话说清楚）
2. **下次怎么避免？**（创建/更新 skill）
3. **方法论有什么改进？**（更新 learning skill）
4. **主人有什么偏好？**（更新 memory）

### 沉淀模式

```
问题是什么 → 怎么解决的 → 学到了什么 → 写入了什么 skill
```

## 快速参考流程图

```
遇到问题
  ↓
Phase 0: 冷静！定义问题
  ↓
Phase 1: 判断问题类型
  ├─ 工具不好用 → 找替代方案 + 记录最佳实践
  ├─ 搜不到 → 换词/换源/分治
  ├─ 不确定 → 对比 + 最小测试
  ├─ 没思路 → 删繁就简 + 分解 + session_search
  ├─ 反复失败 → STOP！回溯假设
  ├─ 文档-代码落差 → 5 层剥洋葱分析
  └─ 自评自判 → 引入第三方裁判
      ↓
Phase 2: 最小验证方案
  ↓
Phase 3: 沉淀为 skill/memory
```

### 一条规则

> **3 次尝试解决不了同一个问题 → 停下来，重新理解问题本身。**
