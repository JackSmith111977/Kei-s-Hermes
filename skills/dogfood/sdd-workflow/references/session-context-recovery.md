# 跨会话上下文恢复协议

> **来源**: 实战教训 — 2026-05-15 SRA EPIC-004 启动时，用户引用前序会话中的「SRA 深度分析完整报告」，session_search 未命中
> **原则**: 诚实 > 猜测 > 沉默。不要假装知道不存在的上下文。

---

## 场景分类

| 场景 | 用户说法示例 | 风险等级 |
|:-----|:------------|:--------:|
| 用户引用前序会话中的「问题/发现」 | 「刚刚发现的问题」「之前讨论过的 Bug」 | 🔴 高 |
| 用户引用前序会话中的「报告/文档」 | 「01:22 的深度分析报告」「上次的架构图」 | 🟡 中 |
| 用户切换项目上下文 | 「回到 sra 项目」「转到 cap-pack」 | 🟡 中 |
| 用户说「继续」但项目不明确 | 「启动 sdd 工作流」（未指定项目） | 🟡 中 |

---

## 恢复协议（4 步）

### Step 1: 会话检索

```python
# 用 session_search 检索前序上下文
session_search(query="SRA 深度分析 完整报告 问题 发现")
session_search(query="EPIC-004 SRA 刚刚 发现")
```

**预期结果**:
- ✅ **命中** → 阅读对应 session 的 summary，提取关键信息，继续 Step 4
- ❌ **未命中** → 进入 Step 2

**已知限制**:
- session_search 基于索引，可能未覆盖所有早期会话
- compaction 可能已清除会话中的具体细节，但 summary 应保留关键信息

### Step 2: 文件系统扫描

当 session_search 未命中时，扫描项目目录寻找可能的文件：

```bash
# 查找最近修改的文档
find ~/projects/<project>/ -name "*.md" -mtime -7 | sort

# 查找包含关键字的文档
grep -rl "深度分析\|完整报告\|EPIC-4\|EPIC-004" ~/projects/<project>/docs/ 2>/dev/null

# 查找非文档目录中的报告
find ~/ -name "*深度*分析*" -o -name "*SRA*完整*" 2>/dev/null | head -10
```

**典型发现**:
- 孤立 Story 文件（如 `SRA-004-01.md`）可能提供线索
- 对话中生成的 HTML/PDF 报告可能在项目根目录或 docs/ 下

### Step 3: 诚实报告（关键！）

当 Step 1 和 Step 2 都未找到时：

```
✅ 正确做法:
「主人，boku 搜索了 session history 和项目文件，
   但找不到「XXX」这个报告/问题的记录喵……
   主人能告诉 boku 具体是什么问题吗？
   或者给个飞书文档链接？😿」

❌ 错误做法:
- 假装找到了并编造内容
- 说「根据报告分析…」但实际上没看到报告
- 沉默跳过，直接开始工作（方向必错）
```

**为什么诚实很重要**:
- 基于错误假设的工作比不工作更糟（方向错了白做）
- 主人宁愿花 30 秒重述问题，也不愿花 30 分钟纠正错误方向
- 诚实暴露了上下文管理的缺口 → 推动系统改进

### Step 4: 基于已知信息重建

如果无法找回原报告/上下文，但项目领域熟悉：

```text
1. 向主人展示项目当前全貌（状态、文件结构、git log）
2. 请主人指明大致方向（如「关于匹配算法的」「关于 CI 的」「关于测试的」）
3. 基于方向重新快速分析（加载对应 skill → 搜索 → 整理）
4. 输出新的分析摘要给主人确认
```

## 项目上下文切换协议

当用户说「回到 XX 项目」「切换到 XX」时：

```text
1. 确认目标项目路径（~projects/<project>/）
2. 检查 git 状态（git status, git log -5）
3. 检查项目文档结构（ls docs/, cat ROADMAP.md）
4. 展示项目全貌
5. 等待主人确认方向，再继续
```

**不要**假设当前项目就是用户想继续的项目。

## 知识沉淀

每次发生跨会话上下文丢失后：
1. 记录丢失的是什么内容 / 什么场景
2. 考虑是否需要在 skill 中增加相关触发词或步骤
3. 如果丢失是因为文件未保存 → 改进保存习惯
4. 如果丢失是因为 session_search 索引延迟 → 记录为已知平台限制
