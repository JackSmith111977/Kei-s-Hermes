# ✅ Self-Review — boku 的自我审查系统

> **核心理念**：在主人批评之前，先自己批评自己。
> **v1.0 诞生**：2026-05-09 文档创作惨痛教训——跳过表格、无 Review、自分析太浅。

---

## 〇、什么时候必加载此 skill？

| 触发场景 | 关键词/动作 | 必需度 |
|:---------|:------------|:------:|
| **创建任何文档** | 创建/更新/飞书/HTML/PDF | 🔴 **强制** |
| **调用新 API** | 第一次使用某 API | 🔴 **强制** |
| **自我分析/调研** | 分析自身/自我认知 | 🔴 **强制** |
| **完成学习任务** | 学习完成/STEP 6 | 🟡 推荐 |
| **发送文件给主人** | 发送/发给/传给 | 🔴 **强制** |
| **任务交付/实施完毕** | 完成/做完了/提交前/汇报前 | 🔴 **强制** |

## 一、通用 Review 清单（所有场景通用）

**每次输出或发送前，逐项检查：**

```markdown
## 📋 通用检查清单

### 🔴 致命项（有一项不通过就不得发送）
- [ ] 是否跳过了困难部分？（表格/复杂结构/边界条件）
- [ ] 是否做了最小测试用例验证？（新 API/新流程）
- [ ] 内容是否完整？（所有章节/所有数据点都在？）
- [ ] 是否有 Review 机制？（自己先检查了一遍？）

### 🟡 重要项
- [ ] 信息是否来自真实来源？（不是凭记忆编的？）
- [ ] 是否有重复内容？（章节之间是否有重叠？）
- [ ] 标题层级是否正确？（h1→h2→h3 不跳级）
- [ ] 链接/引用是否有效？
- [ ] 是否包含敏感信息？（token/key/password？）

### 🟢 质量项
- [ ] 排版是否美观？（留白、配色、对齐）
- [ ] 是否对用户友好？（目录、导航、搜索）
- [ ] 是否可离线/打印？（HTML 打印样式？）
```

## 二、场景 Review 清单

### 场景 A：飞书文档创作

**前置检查（动手前）**：
```markdown
- [ ] 是否先用最小测试验证了 convert+descendant 流程？
- [ ] 表格：是否处理了 merge_info 只读字段？
- [ ] 表格：每个 TableCell 是否有至少一个空 Text 子块？
- [ ] 树结构：是用 children 数组（不是 parent_id）构建的？
- [ ] 分批策略：超过 1000 个 block 时能分批插入吗？
- [ ] 频率限制：编辑操作 3次/秒/文档，是否做了延迟？
```

**产出检查**：
```markdown
- [ ] 所有章节标题都在 raw_content 中能查到？
- [ ] 总字数 ≥ 预期？（经验值：10 章文档 ≥ 5000 字）
- [ ] 表格是否正确渲染？（检查竖线符数量）
- [ ] 没有跳过任何子章节？（如 2.1 → 2.4 中间无遗漏）
```

### 场景 B：HTML/网页文档创作

**前置检查**：
```markdown
- [ ] 是否加载了 visual-aesthetics 和 web-ui-ux-design？
- [ ] 是否先定义了 CSS 设计系统（颜色/间距/字体变量）？
- [ ] 是否有侧边栏导航或多级目录？
- [ ] 是否有打印样式 (@media print)？
- [ ] 是否有响应式设计 (@media max-width)？
```

**产出检查**：
```markdown
- [ ] 用 browser_navigate 打开看过效果？
- [ ] 用 vision_analyze 截图检查过排版？
- [ ] 内容是否与源数据一致？（不是凭记忆写的？）
- [ ] 所有表格渲染正确？（无错位/截断）
- [ ] 代码块语法高亮正常？
- [ ] 侧边栏导航可以点击跳转？
- [ ] 移动端视图正常？
```

### 场景 C：API 集成（飞书/外部服务）

```markdown
- [ ] 先做了最小测试用例（3-5 行数据）？
- [ ] 理解 API 的输入输出格式？
- [ ] 频率限制是否考虑？（退避策略）
- [ ] 错误处理是否完整？（超时/限频/权限不足）
- [ ] 是否有 fallback 方案？（降级策略）
- [ ] token 刷新机制？（2 小时有效期）
- [ ] 是否需要特定权限？（如 docs:permission.member:transfer）
```

### 场景 D：自我分析/系统调查

```markdown
- [ ] 是否用 delegate_task 系统性调查了配置？
- [ ] 是否检查了以下目录？
  - [ ] config.yaml（主配置）
  - [ ] providers/（Provider 配置）
  - [ ] skills/（技能分类和数量）
  - [ ] brain/（三层知识库）
  - [ ] experiences/（经验系统）
  - [ ] learning/（学习记录）
  - [ ] cron/（定时任务）
  - [ ] SOUL.md / MEMORY.md / USER.md
- [ ] 数据是真实调查得到的，不是凭记忆写的？
- [ ] 是否遗漏了"不可见"的部分？（后台进程、定时任务、权限配置）
```

### 场景 E：学习任务完成

```markdown
- [ ] 学习产物是什么？（Skill/文档/笔记）
- [ ] 是否产出了实质性 Artifact？
- [ ] 是否运行了质量门禁（QG）？
- [ ] 是否将经验沉淀到 experiences/ 目录？
- [ ] 是否需要更新 learning skill 的方法论？
- [ ] 是否注册了间隔复习？
```

### 场景 F：🆕 任务交付验证（v1.1.0 新增）

**这是最重要的场景——每次实现完功能、准备向主人汇报前，必须执行此清单。**

**前置检查（动手前）：**
```markdown
- [ ] 是否完全理解了主人的需求？不是「我以为」而是「主人要的」
  - 如果有模糊不清的地方 → 先问清楚再动手
  - 如果主人纠正了方向 → 停下来确认理解一致再继续
- [ ] 实施方案是否已经和主人对齐？（避免方向性返工）
```

**交付前验证（汇报前必做）：**
```markdown
- [ ] 是否做了文档一致性检查？（代码/配置与文档描述是否一致？）
  - Epic 中声称的文件是否都存在？
  - 验收标准是否逐条验证了？
  - 路径/命令是否与实际一致？
- [ ] 是否做了变更范围检查？（只改了该改的？）
- [ ] 是否做了安全检查？（无硬编码密码/token）
- [ ] 是否验证了功能正常运行？（不仅仅是「我觉得可以」）
- [ ] 是否同步更新了所有相关文档？（Epic/README/ROADMAP）
- [ ] 如果涉及 git 提交 → 跑了 commit-quality-check 吗？
- [ ] 如果涉及系统配置 → 验证了配置生效吗？（如 systemctl status）
```

**自动经验沉淀（汇报前执行）：**
当发现以下情况时，说明本次任务产生了可沉淀的经验：
- 发现了新的教训/模式 → 应写入 `experiences/active/`
- 纠正了方向性错误 → 根因分析可复用
- 解决了一个卡住的问题 → 解决方案可复用

```bash
# 沉淀为 L2 经验（替换 --title 和 --content 为实际内容）
python3 ~/.hermes/scripts/knowledge-ingest.py \\
  --type experience --title "本次发现的模式" \\
  --reusability high --confidence 4

# 如涉及核心概念 → 同时沉淀到 L3 Brain
# python3 ~/.hermes/scripts/knowledge-ingest.py \\
#   --type concept --name "概念名" --content "..."
```

**🐌 实战教训：**
- 2026-05-09: 完成 systemd 配置后直接汇报，被问「有做提交前一致性检测吗？」→ 没做。**先验证再汇报，不是先汇报再验证。**
- 2026-05-09: 提出「移植配置文件」方案时没先理解主人真正的需求 → **在提方案前先和主人确认理解一致，而不是假设自己理解对了。**

**🧠 深层根因分析（2026-05-09 需求理解偏差事件）：**

本场景要解决的不是表面错误，而是**认知模式缺陷**。以下是根因链：

```text
问题表象: 手动创建了 systemd 文件后提议「移植到部署仓库」
    ↓
直接原因①: 没有理解项目的核心理念「所有 Agent 应能通过 README 自主安装」
    ↓
直接原因②: 默认假设自己的理解是对的，没停下来确认
    ↓
根因①: 缺少「实施前四问」的自检机制
    ├─ 本方案可被 AI Agent 自主执行吗？→ 否（需要手动复制文件）
    ├─ 符合项目设计哲学吗？→ 否（「移植文件」违背「脚本生成」理念）
    ├─ 跨环境可重现吗？→ 否（路径硬编码）
    └─ 最小化人工干预吗？→ 否（需要用户手动操作）
    ↓
根因②: 重功能实现、轻方案设计
    └─ 只考虑「怎么让这东西跑起来」，没思考「这东西属于项目的哪部分」
    ↓
根因③: 没有站在「消费者」视角思考
    └─ 如果另一个 AI Agent 读这个项目的文档，它能自主完成部署吗？
```

**工程化预防措施：**

| 措施 | 类型 | 执行时机 |
|:---|:---|:---:|
| **实施前四问自检**（见下方模板） | 🔴 流程门禁 | 任何实施开始前 |
| **先对齐再动手** — 复杂方案先输出设计摘要给主人确认 | 🔴 流程门禁 | 调研→实施之间 |
| **消费者视角测试** — 想象自己是另一个 AI Agent，按自己的方案做一遍 | 🟡 心理模型 | 方案设计时 |
| **用脚投票** — 如果方案中有任何「手动步骤」，必须自动化 | 🟡 质量标准 | 方案评审时 |

**实施前四问模板（每次开始实施前逐项回答）：**

```markdown
### 🔍 实施前四问

1. [ ] 这个方案可以被 AI Agent 阅读文档后自主执行吗？
   → 如果答案是「不行，需要手动XX」→ ❌ 需要重新设计

2. [ ] 这个方案符合项目的核心设计哲学吗？
   → 如果不确定 → 问主人确认

3. [ ] 这个方案在另一台全新机器上也能重现吗？
   → 如果依赖特定路径/环境 → ❌ 需要抽象化

4. [ ] 我理解的需求和主人真正的需求一致吗？
   → 如果不确定 → 先输出设计确认再实施
```

**🔴 铁律：如果「实施前四问」有一项不通过，必须先停下并和主人对齐，不能继续实施。**

## 三、自我进化机制 ⭐

**核心**：每次犯错或被纠正后，立即更新此 skill，让它越来越强。

### 进化规则

```text
当以下情况发生时 → 必须更新 self-review：
  1. 主人指出某个错误 → 新增对应的检查项到对应场景
  2. 发现了新的问题类型 → 新增检查场景或检查项
  3. 学会了新的最佳实践 → 更新现有检查项为更优做法
  4. 新增了能力/工具 → 新增检查场景
```

### 进化记录

| 日期 | 版本 | 变更 | 触发事件 |
|:----|:----:|:-----|:---------|
| 2026-05-09 | v1.0 | 初始创建 | 文档创作跳过表格+无 Review+自分析太浅 |
| 2026-05-09 | v1.1 | 新增「任务交付验证」场景 F + 铁律「先验证再汇报」+ 扩展触发器 | 完成 systemd 配置后未做一致性检测被指出 + 需求理解偏差被纠正 |
| 2026-05-09 | v1.2 | 新增「深层根因分析」+ 「工程化预防措施」+ 「实施前四问」模板 | 提出「移植配置文件」方案暴露了认知模式缺陷——重执行轻设计、缺少前置对齐、缺少消费者视角 |

### 检查项老化机制

- 如果某个检查项连续 3 次都 pass → 降级为 🟢（高频通过说明已形成习惯）
- 如果某个检查项连续 2 次 fail → 升级为 🔴（说明这块是薄弱环节）
- 如果超过 30 天没有触发某场景 → 自动复习该场景的检查项

## 四、快速执行流程

当触发此 skill 时，按以下流程执行：

```text
1. 判断当前场景（文档/API/自分析/学习/通用/任务交付）
2. 加载对应场景的 Review 清单
3. 逐项检查（用 terminal/read_file/vision 验证）
4. 记录检查结果：
   - 全部通过 → ✅ 可以发送
   - 有 🔴 失败 → 🛑 禁止发送，修复后重检
   - 只有 🟡/🟢 失败 → ⚠️ 修复后发送，记录改进点
5. 如果发现新的错误类型 → 更新此 skill
6. 🔴 铁律：**任何「做完了」的汇报前，必须先跑 self-review。先验证再汇报，不是先汇报再被检查。**
```
