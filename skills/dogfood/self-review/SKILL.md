# ✅ Self-Review — boku 的自我审查系统

> **核心理念**：在主人批评之前，先自己批评自己。
> **v1.0 诞生**：2026-05-09 文档创作惨痛教训——跳过表格、无 Review、自分析太浅。

---

## 〇、什么时候必加载此 skill？

| 触发场景 | 关键词/动作 | 必需度 |
|:---------|:------------|:------:|
| **创建任何文档** | 创建/更新/飞书/HTML/PDF | 🔴 **强制** |
| **调用新 API** | 第一次使用某 API | 🔴 **强制** |
| **自我分析/调研** | 分析自身/自我认知 | 🔴 **强制** |
| **完成学习任务** | 学习完成/STEP 6 | 🟡 推荐 |
| **发送文件给主人** | 发送/发给/传给 | 🔴 **强制** |
| **任务交付/实施完毕** | 完成/做完了/提交前/汇报前 | 🔴 **强制** |

## 一、通用 Review 清单（所有场景通用）

**每次输出或发送前，逐项检查：**

```markdown
## 📋 通用检查清单

### 🔴 致命项（有一项不通过就不得发送）
- [ ] 是否跳过了困难部分？（表格/复杂结构/边界条件）
- [ ] 是否做了最小测试用例验证？（新 API/新流程）
- [ ] 内容是否完整？（所有章节/所有数据点都在？）
- [ ] 是否有 Review 机制？（自己先检查了一遍？）

### 🟡 重要项
- [ ] 信息是否来自真实来源？（不是凭记忆编的？）
- [ ] 是否有重复内容？（章节之间是否有重叠？）
- [ ] 标题层级是否正确？（h1→h2→h3 不跳级）
- [ ] 链接/引用是否有效？
- [ ] 是否包含敏感信息？（token/key/password？）

### 🟢 质量项
- [ ] 排版是否美观？（留白、配色、对齐）
- [ ] 是否对用户友好？（目录、导航、搜索）
- [ ] 是否可离线/打印？（HTML 打印样式？）
```

## 二、场景 Review 清单

### 场景 A：飞书文档创作

**前置检查（动手前）**：
```markdown
- [ ] 是否先用最小测试验证了 convert+descendant 流程？
- [ ] 表格：是否处理了 merge_info 只读字段？
- [ ] 表格：每个 TableCell 是否有至少一个空 Text 子块？
- [ ] 树结构：是用 children 数组（不是 parent_id）构建的？
- [ ] 分批策略：超过 1000 个 block 时能分批插入吗？
- [ ] 频率限制：编辑操作 3次/秒/文档，是否做了延迟？
```

**产出检查**：
```markdown
- [ ] 所有章节标题都在 raw_content 中能查到？
- [ ] 总字数 ≥ 预期？（经验值：10 章文档 ≥ 5000 字）
- [ ] 表格是否正确渲染？（检查竖线符数量）
- [ ] 没有跳过任何子章节？（如 2.1 → 2.4 中间无遗漏）
```

```markdown
- [ ] 是否加载了 visual-aesthetics 和 web-ui-ux-design？

- [ ] 是否有侧边栏导航或多级目录？
- [ ] 是否有打印样式 (@media print)？
- [ ] 是否有响应式设计 (@media max-width)？
```

**产出检查**：
```markdown
- [ ] 用 browser_navigate 打开看过效果？
- [ ] 用 vision_analyze 截图检查过排版？
- [ ] 内容是否与源数据一致？（不是凭记忆写的？）
- [ ] 所有表格渲染正确？（无错位/截断）
- [ ] 代码块语法高亮正常？
- [ ] 侧边栏导航可以点击跳转？
- [ ] 移动端视图正常？
```

```markdown
- [ ] 先做了最小测试用例（3-5 行数据）？
- [ ] 理解 API 的输入输出格式？
- [ ] 频率限制是否考虑？（退避策略）
- [ ] 错误处理是否完整？（超时/限频/权限不足）
- [ ] 是否有 fallback 方案？（降级策略）

```

```markdown
- [ ] 是否用 delegate_task 系统性调查了配置？
- [ ] 是否检查了以下目录？
  - [ ] config.yaml（主配置）
  - [ ] providers/（Provider 配置）
  - [ ] skills/（技能分类和数量）
  - [ ] brain/（三层知识库）
  - [ ] experiences/（经验系统）
  - [ ] learning/（学习记录）
  - [ ] cron/（定时任务）
  - [ ] SOUL.md / MEMORY.md / USER.md
- [ ] 数据是真实调查得到的，不是凭记忆写的？
- [ ] 是否遗漏了"不可见"的部分？（后台进程、定时任务、权限配置）
```

### 场景 E：学习任务完成

```markdown
- [ ] 学习产物是什么？（Skill/文档/笔记）
- [ ] 是否产出了实质性 Artifact？
- [ ] 是否运行了质量门禁（QG）？
- [ ] 是否将经验沉淀到 experiences/ 目录？
- [ ] 是否需要更新 learning skill 的方法论？
- [ ] 是否注册了间隔复习？
```

**🌀 三阶段行动检查（v1.4.0 新增）**:

学习完成后，额外检查：

```markdown
- [ ] **学习产出是「系统改进」还是「单点知识」？**
  - 系统改进（跨模块/跨流程）→ 必须走三阶段（学习→基建→修复）
  - 单点知识（一个概念/一个函数）→ 正常沉淀即可
- [ ] **是否产出可执行的工作流？**
  - 不只是思想/概念/SKILL.md，要有脚本/门禁/可执行检查
  - 如果学习产出了方法论但没有对应的可执行工作流 → 学习不够工程化
- [ ] **是否建立了机械门禁？**
  - 防止同类问题再次发生
  - 门禁三要素：可自动执行 + 结果确定 + 无法绕开
- [ ] **主人偏好提醒**：主人说过「不只是建立思想，更要建立可用的工作流skill」
  - 如果学习成果是一个方法论 → 问自己：它能变成一个独立的 skill 吗？
```

### 场景 G：🧠 文件/函数存在性验证（v1.3.0 新增 — 防幻觉）

> **触发条件**: 在引用任何文件、函数、类、或方法之前，必须执行此检查。
> **为什么需要独立场景**: AI agent 经常「记得」文件/函数存在，但实际不存在。
> 这比写错代码更危险——它会误导整个分析方向。
>
> **实战案例 (2026-05-11)**: boku 声称 SRA 项目有 `tests/contract_formatter.py` 含 `skill_yaml_fixture()`，
> 但此文件**从未在 git 历史中存在过**。完全幻觉。根因：凭「记忆」而非 `ls`/`grep` 验证。

**🔴 铁律：在任何引用操作之前，必须先用 terminal 验证存在性。**

```markdown
### 🧠 文件/函数存在性验证清单

#### 文件引用
- [ ] 引用文件前是否先跑了 `read_file` 或 `ls` 确认？
  - ❌ 错误：  "在 sra/tests/contract_formatter.py 中有 skill_yaml_fixture()"
  - ✅ 正确：  "boku 先 grep -rl 'skill_yaml_fixture' 确认文件存在"
- [ ] 如果用的是 glob 模式（`**/*.py`），确认匹配结果不为空？
- [ ] 如果文件在特定 git 分支上，确认当前分支包含该文件？

#### 函数/类引用
- [ ] 引用前是否先 `grep -rn "function_name"` 确认定义存在？
- [ ] 如果函数是「新增的」（刚在 session 中创建），确认路径和签名正确？
- [ ] 是否把「计划要写的函数」当成了「已经存在的函数」？

#### 测试数据引用
- [ ] 引用 `tests/fixtures/` 数据前，是否先 `ls tests/fixtures/` 确认目录结构？
- [ ] 引用测试 fixture 模式前，是否先 `grep -rn "FIXTURES_DIR" tests/` 确认已有模式？

#### 记住的三大幻觉陷阱
1. 🪤 **「我好像记得有」** → 不要信记忆，信 `find`/`grep`
2. 🪤 **「之前 session 里有」** → 上下文压缩可能丢弃了文件信息，重新验证
3. 🪤 **「刚创建的」** → 确认 `write_file` 的返回状态，不是「记得刚才写过」
```

### 场景 F：🆕 任务交付验证（v1.1.0 新增）

**这是最重要的场景——每次实现完功能、准备向主人汇报前，必须执行此清单。**

**前置检查（动手前）：**
```markdown
- [ ] 是否完全理解了主人的需求？不是「我以为」而是「主人要的」
  - 如果有模糊不清的地方 → 先问清楚再动手
  - 如果主人纠正了方向 → 停下来确认理解一致再继续
- [ ] 实施方案是否已经和主人对齐？（避免方向性返工）
- [ ] **是否做了「测试模式发现」？**（写新测试前必须扫已有 fixture）
  - 命令: `grep -rn "FIXTURES_DIR\|fixtures/" tests/ 2>/dev/null | head -10`
  - 如果有 `tests/fixtures/` 目录，新测试**必须**优先使用它
  - 🔴 **禁止引入 `~/.hermes/skills` 等运行时依赖**（CI 中没有）
  - 🚫 **不要幻觉文件/函数存在** — 引用前先 `read_file` 或 `grep -rl` 确认
```

**交付前验证（汇报前必做）：**\n```markdown\n- [ ] 是否做了文档一致性检查？（代码/配置与文档描述是否一致？）\n  - Epic 中声称的文件是否都存在？\n  - 验收标准是否逐条验证了？\n  - 路径/命令是否与实际一致？\n- [ ] 是否做了变更范围检查？（只改了该改的？）\n- [ ] 是否做了安全检查？（无硬编码密码/token）\n- [ ] 是否验证了功能正常运行？（不仅仅是「我觉得可以」）\n- [ ] 是否同步更新了所有相关文档？（Epic/README/ROADMAP）\n- [ ] 如果涉及 git 提交 → 跑了 commit-quality-check 吗？\n- [ ] 如果涉及系统配置 → 验证了配置生效吗？（如 systemctl status）\n\n**🔄 CI 流水线验证（新增 — 2026-05-11 实战教训）**\n```markdown\n- [ ] **本地 lint 是否通过？**（ruff check / flake8 / eslint）\n  - 本地测试通过 ≠ CI 能过。lint 问题是最常见的 CI 失败原因\n- [ ] **是否做了 lint 检查？**\n  - 命令: `ruff check .` 或 `ruff check skill_advisor/ tests/`\n- [ ] **是否 mock 了 CI 环境差异？**\n  - 测试是否依赖了 CI 中没有的目录/服务？（如 ~/.hermes/skills）\n  - 如果有 → 用 fixture 数据替代或 pytest.skip() 降级\n- [ ] **提交后是否确认了 CI 状态？**\n  - `gh run list --limit 1` 确认 workflow 通过\n  - 如果失败 → 先修再继续，不要等主人指出\n```\n\n**📌 实战教训（2026-05-11）**:\n- 本地 `pytest 290 passed` 但 CI 中 ruff `I001` import sorting 失败\n- 根因: 本地忘了跑 `ruff check`。CI 门禁抓住了本地漏掉的 lint 问题\n- **教训**: 提交前必须 `ruff check .` + `pytest -q` 双重验证

**自动经验沉淀（汇报前执行）：**
当发现以下情况时，说明本次任务产生了可沉淀的经验：
- 发现了新的教训/模式 → 应写入 `experiences/active/`
- 纠正了方向性错误 → 根因分析可复用
- 解决了一个卡住的问题 → 解决方案可复用

```bash
# 沉淀为 L2 经验（替换 --title 和 --content 为实际内容）
python3 ~/.hermes/scripts/knowledge-ingest.py \\
  --type experience --title "本次发现的模式" \\
  --reusability high --confidence 4

# 如涉及核心概念 → 同时沉淀到 L3 Brain
# python3 ~/.hermes/scripts/knowledge-ingest.py \\
#   --type concept --name "概念名" --content "..."
```

**🐌 实战教训：**
- 2026-05-09: 完成 systemd 配置后直接汇报，被问「有做提交前一致性检测吗？」→ 没做。**先验证再汇报，不是先汇报再验证。**
- 2026-05-09: 提出「移植配置文件」方案时没先理解主人真正的需求 → **在提方案前先和主人确认理解一致，而不是假设自己理解对了。**
- 2026-05-11: 开发 SRA `test_contract.py` 时使用了 `~/.hermes/skills` 而非已有 `tests/fixtures/` 数据 → CI 失败。**眼睛看到 `test_matcher.py` 的 `FIXTURES_DIR` 模式，但思维没有复用。** 需要显式的「测试模式发现」步骤来打破这个认知盲区。

**🧠 深层根因分析（2026-05-09 需求理解偏差事件）：**

本场景要解决的不是表面错误，而是**认知模式缺陷**。以下是根因链：

```text
问题表象: 手动创建了 systemd 文件后提议「移植到部署仓库」
    ↓
直接原因①: 没有理解项目的核心理念「所有 Agent 应能通过 README 自主安装」
    ↓
直接原因②: 默认假设自己的理解是对的，没停下来确认
    ↓
根因①: 缺少「实施前四问」的自检机制
    ├─ 本方案可被 AI Agent 自主执行吗？→ 否（需要手动复制文件）
    ├─ 符合项目设计哲学吗？→ 否（「移植文件」违背「脚本生成」理念）
    ├─ 跨环境可重现吗？→ 否（路径硬编码）
    └─ 最小化人工干预吗？→ 否（需要用户手动操作）
    ↓
根因②: 重功能实现、轻方案设计
    └─ 只考虑「怎么让这东西跑起来」，没思考「这东西属于项目的哪部分」
    ↓
根因③: 没有站在「消费者」视角思考
    └─ 如果另一个 AI Agent 读这个项目的文档，它能自主完成部署吗？
```

**工程化预防措施：**

| 措施 | 类型 | 执行时机 |
|:---|:---|:---:|
| **实施前四问自检**（见下方模板） | 🔴 流程门禁 | 任何实施开始前 |
| **先对齐再动手** — 复杂方案先输出设计摘要给主人确认 | 🔴 流程门禁 | 调研→实施之间 |
| **消费者视角测试** — 想象自己是另一个 AI Agent，按自己的方案做一遍 | 🟡 心理模型 | 方案设计时 |
| **用脚投票** — 如果方案中有任何「手动步骤」，必须自动化 | 🟡 质量标准 | 方案评审时 |

**实施前五问模板（每次开始实施前逐项回答）：**

```markdown
### 🔍 实施前五问

1. [ ] 这个方案可以被 AI Agent 阅读文档后自主执行吗？
   → 如果答案是「不行，需要手动XX」→ ❌ 需要重新设计

2. [ ] 这个方案符合项目的核心设计哲学吗？
   → 如果不确定 → 问主人确认

3. [ ] 这个方案在另一台全新机器上也能重现吗？
   → 如果依赖特定路径/环境 → ❌ 需要抽象化

4. [ ] 我理解的需求和主人真正的需求一致吗？
   → 如果不确定 → 先输出设计确认再实施

5. [ ] **是否检查了已有的测试基础设施和测试数据？**
   → 命令: `grep -rn "FIXTURES_DIR\|fixtures/" tests/ 2>/dev/null | head -10`
   → 如果有 `tests/fixtures/` 目录，优先复用而非新建
   → 如果答案是否 → 先扫一遍 fixture 目录再设计测试
   → 🔴 这是 2026-05-11 的惨痛教训：**眼睛看到 ≠ 思维复用**。test_matcher.py 已经用了 FIXTURES_DIR，但 test_contract.py 没用。
```

**🔴 铁律：如果「实施前五问」有一项不通过，必须先停下并和主人对齐，不能继续实施。**

## 三、自我进化机制 ⭐

**核心**：每次犯错或被纠正后，立即更新此 skill，让它越来越强。

### 进化规则

```text
当以下情况发生时 → 必须更新 self-review：
  1. 主人指出某个错误 → 新增对应的检查项到对应场景
  2. 发现了新的问题类型 → 新增检查场景或检查项
  3. 学会了新的最佳实践 → 更新现有检查项为更优做法
  4. 新增了能力/工具 → 新增检查场景
```

### 检查项老化机制

- 如果某个检查项连续 3 次都 pass → 降级为 🟢（高频通过说明已形成习惯）
- 如果某个检查项连续 2 次 fail → 升级为 🔴（说明这块是薄弱环节）
- 如果超过 30 天没有触发某场景 → 自动复习该场景的检查项

## 四、快速执行流程

当触发此 skill 时，按以下流程执行：

```text
1. 判断当前场景（文档/API/自分析/学习/通用/任务交付）
2. 加载对应场景的 Review 清单
3. 逐项检查（用 terminal/read_file/vision 验证）
4. 记录检查结果：
   - 全部通过 → ✅ 可以发送
   - 有 🔴 失败 → 🛑 禁止发送，修复后重检
   - 只有 🟡/🟢 失败 → ⚠️ 修复后发送，记录改进点
5. 如果发现新的错误类型 → 更新此 skill
6. 🔴 铁律：**任何「做完了」的汇报前，必须先跑 self-review。先验证再汇报，不是先汇报再被检查。**
```
