# 系统性改造实战案例 — SRA 文档报告生命周期改造

> **日期**: 2026-05-11
> **场景**: 从手写 HTML 升级为数据驱动报告系统
> **结果**: doc-alignment v2.x → v3.0，SRA 项目验证通过

## 改造流程（可复用模式）

### Phase 0: 深度分析

```
① 加载当前 skill → 完整阅读现有工作流
② 识别痛点：手写HTML易遗漏、数据源分散、无验证
③ 定位根因：缺少自动化工具和标准数据格式
```

### Phase 1: 架构设计

```
④ 设计方案 → 数据驱动 JSON → HTML 渲染器
⑤ 明确边界：哪些自动化(生成/验证)，哪些人工(JSON字段)
⑥ 确定位置：生成器放 ~/.hermes/scripts/ 跨项目共享
```

### Phase 2: 核心构建

```
⑦ 先造工具：零依赖生成器 generate-project-report.py
⑧ 再定格式：project-report.json 标准 Schema
⑨ 后加验证：--verify 漂移检测模式
```

原子顺序：工具 → 数据 → 验证（每一步可独立测试）

### Phase 3: 工作流升级

```
⑩ 更新 skill：doc-alignment v3.0，全生命周期
⑪ 添加模板：新项目可复用的 project-report-template.json
⑫ 添加实战案例：本文档
```

### Phase 4: 验证 & 通用化

```
⑬ 在真实项目(SRA)跑完整流程
⑭ --verify 确认零漂移
⑮ 确认生成器位置 ~/.hermes/scripts/ 跨项目可用
```

## 关键设计决策

| 决策 | 选项 | 选择 | 原因 |
|:-----|:-----|:-----|:------|
| 模板引擎 | Jinja2 vs f-string | f-string + 函数 | 零依赖，所有环境可用 |
| 存放位置 | 各项目独立 vs 共享 | `~/.hermes/scripts/` | 改一次所有项目受益 |
| CSS | 外链CDN vs 内嵌 | 内嵌CSS | 离线可打开，无网络依赖 |
| 验证模式 | 独立脚本 vs 内置 | `--verify` 参数 | 一体工具，减少心智负担 |
| 数据格式 | YAML vs JSON | JSON | Python原生支持，无需pyyaml |

## 反模式

| 反模式 | 表现 | 正确做法 |
|:-------|:-----|:---------|
| 先写文档再建工具 | 先改HTML再考虑自动化 | 先建生成器再生成HTML |
| 一次改所有项目 | 同时改多个项目的HTML | 先在SRA验证，再模板化 |
| 完美主义 | 功能齐全才上线 | MVP先跑通核心链路再扩展 |
