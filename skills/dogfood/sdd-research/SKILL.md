# 🔬 sdd-research — SDD RESEARCH 阶段方法论

> **定位**: SDD 工作流的 RESEARCH 阶段专用调研方法论。
> **复用**: deep-research 的核心方法论（多轮搜索、来源分级、MECE、对比矩阵）。
> **区别**: 产出为**调研笔记（.md）** 而非 PDF/PPT，直接输入 SPEC 的 CREATE 阶段。
> **前置条件**: SDD CLARIFY 阶段已完成（需求范围已确认）。

---

## 〇、SDD 中的位置

```
SDD 全流程:
CLARIFY ──→ RESEARCH ──→ CREATE ──→ QA_GATE ──→ REVIEW ──→ APPROVED
 需求澄清     👆 本 skill     写 Spec      质量检查     主人审阅     批准
              sdd-research
```

**RESEARCH 阶段的目标**: 为 SPEC 创作收集足够的技术信息，产出结构化调研笔记。
**RESEARCH 阶段的产出**: `docs/research/{spec-id}.md` — 调研笔记文档。

---

## 一、RESEARCH 流程总览

```
Phase 0: 范围对齐
    ↓
Phase 1: 多轮搜索（广度 → 深度 → 精炼）
    ↓
Phase 2: 来源分级与交叉验证
    ↓
Phase 3: 结构化分析（MECE + 对比矩阵 + 决策建议）
    ↓
Phase 4: 产出调研笔记 → 输入 SPEC CREATE
```

## Phase 0: 范围对齐

**在开始搜索前，先对齐 CLARIFY 阶段的产出：**

| 问题 | 作用 | 来自 CLARIFY 的输入 |
|:-----|:-----|:--------------------|
| 这个 SPEC 要回答什么问题？ | 定义调研边界 | 用户故事、核心问题 |
| 需要调研哪些子主题？ | 拆分为 3-5 个方向 | AC 列表、范围矩阵 |
| 已有信息 vs 需要调研的？ | 避免重复工作 | 已有 Epic、已有 Spec |
| 调研深度要求？ | P0/P1/P2 决定搜索轮次 | SPEC 优先级 |

**拆分原则（复用 deep-research SCQA 框架）**：

| 元素 | 问题 | 调研场景示例 |
|:-----|:------|:------------|
| **S**ituation 现状 | 当前是什么情况？ | HermesAdapter 用 SKILLS/{id}/ 目录复制 |
| **C**omplication 挑战 | 有什么问题？ | install.scripts 等配置未消费 |
| **Q**uestion 问题 | 要解决什么？ | 如何让安装引擎完整消费 cap-pack.yaml |
| **A**nswer 答案（待调研） | 可能的方案？ | 扩展现有 _install_skills / 新增方法 |

## Phase 1: 多轮搜索

### 搜索轮次规则

| 轮次 | 目的 | 最少搜索数 | SDD 场景 |
|:----:|:-----|:----------:|:---------|
| 第 1 轮 🌐 | **广度扫描** — 覆盖全景 | 3-5 个搜索 | 识别所有候选方案/技术/库 |
| 第 2 轮 🔍 | **深度挖掘** — 核心细节 | 2-3 个搜索 | 逐一验证候选方案的关键维度 |
| 第 3 轮 ✨ | **精炼验证** — 查漏补缺 | 1-2 个搜索 | 验证异议、补充遗漏角度 |

**铁律**：
- ❌ 禁止只做 1 轮搜索就下结论
- ❌ 禁止第 2 轮搜索又重新开始（必须基于第 1 轮发现深入）
- ✅ 每个轮次的发现标注轮次编号（Round 1/2/3）

### SDD 特化：代码级调研

SDD RESEARCH 与一般调研的不同在于：**需要检查现有代码**。

```
调研类型            方法                       适用场景
──────────────────────────────────────────────────────────
🔍 代码级调研       read_file + grep           现有实现分析
🌐 外部调研         web_search + web_extract   技术选型/第三方方案
📊 数据调研         search_files + 统计         项目现状摸底
💬 历史调研         session_search + git log    决策追溯
```

**SDD RESEARCH 建议混合使用**：先代码级调研摸清现状，再外部调研找方案。

## Phase 2: 来源分级与交叉验证

### 来源分级

| 等级 | 来源类型 | SDD 场景示例 | 信任度 |
|:-----|:---------|:------------|:------:|
| 🥇 | 项目自身代码/文档 | 当前代码库、已有 Epic/Spec | ✅ 最高 |
| 🥇 | 官方文档/API 文档 | GitHub README、API Docs | ✅ 高 |
| 🥈 | 技术博客/社区 | Medium、Stack Overflow | ⚠️ 需交叉验证 |
| 🥉 | AI 聚合 | 无原文链接的内容 | ❌ 仅作线索 |

### 交叉验证标准（复用 deep-research）

| 数据类型 | 验证要求 | SDD 示例 |
|:---------|:---------|:---------|
| **核心结论** | ≥2 个独立来源 | 技术选型结论需代码验证 + 文档佐证 |
| **API 接口** | 官方文档 + 代码调用示例 | 确认 Hermes API 签名 |
| **配置格式** | 官方文档 + 项目已有配置 | 确认 YAML 结构 |
| **性能数据** | 基准测试 + 实际运行验证 | 不做纯理论假设 |

## Phase 3: 结构化分析

### MECE 分类（复用 deep-research）

将所有方案/发现按 MECE 原则分类：

```
所有方案
├── 方案类 A（特征 A）
│   ├── 具体方案 1
│   └── 具体方案 2
├── 方案类 B（特征 B）
└── 方案类 C（兜底）
```

### SDD 特化：多维对比矩阵

| 维度 | 方案 A | 方案 B | 方案 C |
|:-----|:------:|:------:|:------:|
| **实现成本** | 低/中/高 | 低/中/高 | 低/中/高 |
| **向前兼容** | ✅/⚠️/❌ | ✅/⚠️/❌ | ✅/⚠️/❌ |
| **维护成本** | 低/中/高 | 低/中/高 | 低/中/高 |
| **风险等级** | 🟢/🟡/🔴 | 🟢/🟡/🔴 | 🟢/🟡/🔴 |
| **推荐度** | ⭐⭐⭐ | ⭐⭐ | ⭐ |

### ADR 记录（Architecture Decision Record）

调研过程中形成的决策应记录为 ADR：

```markdown
### ADR-{SPEC}-{seq}: 决策标题

**状态**: proposed | accepted | deprecated
**日期**: YYYY-MM-DD

**背景**: 为什么要做这个决策？

**方案对比**:
- 方案 A: 优点/缺点
- 方案 B: 优点/缺点

**决策**: 选择了方案 A

**理由**: 因为 X, Y, Z

**后果**: 选择了方案 A 意味着...
```

## 三、SDD RESEARCH 检查清单

### 调研前
- [ ] CLARIFY 阶段已完成（需求范围已确认）
- [ ] 拆分了 3-5 个子主题
- [ ] 明确了已有信息 vs 需要调研的
- [ ] 确定了调研深度（P0/P1/P2）

### 调研中
- [ ] 至少完成了 2 轮搜索
- [ ] 代码级调研 + 外部调研混合使用
- [ ] 核心结论有 ≥2 个来源交叉验证
- [ ] 保存了搜索记录（标注轮次）
- [ ] 记录了来源 URL

### 调研后
- [ ] MECE 分类完成
- [ ] 对比矩阵完成（≥4 维度）
- [ ] ADR 记录完成
- [ ] 调研笔记已输出到 `docs/research/{spec-id}.md`
- [ ] 待决策点已标识
- [ ] 建议方向已给出

## 五、常见陷阱

| 陷阱 | 表现 | 避免方法 |
|:-----|:------|:---------|
| **跳过广度扫描** | 直接深挖一个方案 | 强制第 1 轮至少搜索 3 个方向 |
| **交叉验证不足** | 核心结论只有一个来源 | 每条核心发现标注来源等级 |
| **调研笔记不输出** | 调研结果只存在于对话中 | 强制输出到 `docs/research/{spec-id}.md` |
| **ADR 缺失** | 决策理由不可追溯 | 每个方案对比记录 ADR |
| **与 CLARIFY 脱节** | 调研方向偏离需求范围 | Phase 0 对齐后再搜索 |
