# Skill Creator 执行指南
> 详细的操作步骤和决策参考

**版本：** 1.0.0 | **更新：** 2026-03-30

---

## 前置条件

在开始 Skill 创建流程之前，确认以下条件：

1. **明确的任务需求** — 用户提出了创建/优化/评估 Skill 的请求
2. **目标 Skill 名称** — 小写字母 + 连字符格式（如 `code-review`）
3. **输出位置** — 项目级（`.claude/skills/`）或全局级（`~/.hermes/skills/`）

---

## 创建新 Skill 完整流程

### Step 1：需求澄清

向用户提出 5 个关键问题并收集回答。

**提问模板：**

```
📋 收到 Skill 创建请求！

请回答以下问题帮助我设计最合适的 Skill：

1. **主要用途是什么？**
   - [根据用户初步描述给出类型建议]

2. **触发方式偏好？**
   - 显式命令（如 /skill-name）
   - 隐式触发词（如"帮我审查代码"）
   - 两者都要

3. **目标使用者？**
   - 个人使用
   - 团队共享

4. **是否涉及外部工具？**
   - 脚本执行
   - API 调用
   - 无外部依赖

5. **期望的输出格式？**
   - Markdown 报告
   - 代码文件
   - JSON 数据
   - 其他
```

### Step 2：用例定义

收集 2-3 个具体用例，确认触发方式。

**用例收集模板：**

```
📋 请提供 2-3 个具体使用场景：

**用例 1：**
- 触发条件：[用户说什么/做什么]
- 执行步骤：[1 → 2 → 3]
- 预期结果：[输出格式和内容]

**用例 2：**
...

**用例 3（可选）：**
...
```

### Step 3：选择类型 + 模式

根据 Step 1-2 的结果，从 9 种类型中选择最匹配的。

**决策参考：**

| 用户说的 | 推荐类型 |
|----------|----------|
| "教我怎么用 XX 库" | 类型1: 库和API参考 |
| "验证 XX 功能是否正常" | 类型2: 产品验证 |
| "查询 XX 数据" | 类型3: 数据获取与分析 |
| "根据 XX 生成代码" | 类型4: 代码生成 |
| "生成 PR 描述/文档" | 类型5: 文档生成 |
| "审查代码安全/质量" | 类型6: 代码审查 |
| "部署到生产环境" | 类型7: 部署流程 |
| "统一代码风格" | 类型8: 团队规范 |
| "调研 XX 技术" | 类型9: 调研整理 |

### Step 4：生成草稿

1. 从 `templates/` 加载对应模板
2. 将用例信息填入模板
3. 生成变更摘要预览

**输出格式：**

```markdown
## 📋 Skill 草稿预览

**类型：** [类型名称]
**设计模式：** [模式组合]
**触发命令：** /[skill-name]
**触发词：** [词1, 词2, 词3]

### 变更摘要
- **新增：** [新增的章节/内容]
- **基于模板：** [模板文件名]

### 核心工作流程
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

### 下一步
确认后我将写入文件。是否需要修改？
```

### Step 5：用户确认

等待用户确认或修改。如需要修改：
- 记录修改意见
- 返回 Step 4 重新生成
- 再次确认

### Step 6：写入文件

1. 创建目录结构
2. 写入 SKILL.md
3. 写入辅助文件（模板/参考/检查清单/脚本）

### Step 7：Review 检查

逐项检查创建检查清单中的必须项。

### Step 8：测试建议

生成 3 类测试用例，建议用户执行验证。

## 优化已有 Skill 流程

### 触发条件
- 用户明确说"优化 XX skill"
- Review 检查发现严重问题
- 评估结果显示通过率低

### 优化维度

#### 1. 精简主文档
- 目标：SKILL.md < 500 行
- 方法：将详细内容移至 `references/`
- 保留：核心指令 + 快速参考

#### 2. 消除重复
- 检查重复章节
- 合并相似内容
- 使用引用链接

#### 3. 补充缺失
- 踩坑清单（Gotchas）
- 示例（好坏对比）
- 检查清单

#### 4. 更新触发词
- 基于实际使用情况
- 增加同义词
- 移除不准确的触发词

## 常见问题

### Q1：如何判断 Skill 类型？
优先看用户的核心动词：
- "用/调用/参考" → 类型1
- "验证/测试/检查" → 类型2 或 类型6
- "查询/分析/报表" → 类型3
- "生成代码" → 类型4
- "生成文档" → 类型5
- "部署/发布" → 类型7
- "规范/风格" → 类型8
- "调研/整理" → 类型9

### Q2：如何处理多类型混合？
选择主要类型，在其他类型的核心章节中引用。例如"代码审查 + 部署"：
- 主类型：代码审查（类型6）
- 在"安全部署"章节引用部署流程（类型7）

### Q3：Pipeline 模式何时使用？
当 Skill 需要：
- 严格顺序执行
- 中间状态追踪
- 用户检查点确认
- 失败时回滚

适用类型：产品验证（类型2）、代码生成（类型4）、部署流程（类型7）

### Q4：如何编写好的期望（Expectations）？
- ✅ "输出包含 X 章节"
- ✅ "报告包含至少 3 个代码示例"
- ❌ "输出质量好"
- ❌ "结果令人满意"

原则：具体、可验证、可区分

### Q5：主文档多长合适？
- 目标：< 500 行
- 理想：200-350 行
- 如果 > 500 行：将详细内容移至 `references/`

## 版本管理规范

### 版本号格式
`主版本号.次版本号.修订号`（如 1.2.3）

### 更新规则
| 变更类型 | 版本更新 | 示例 |
|----------|----------|------|
| 不兼容的修改 | 主版本号 +1 | 1.x.x → 2.0.0 |
| 向后兼容的新功能 | 次版本号 +1 | 1.0.x → 1.1.0 |
| 向后兼容的问题修复 | 修订号 +1 | 1.0.0 → 1.0.1 |

### 更新记录格式

```markdown
## 更新记录

### v1.2.0 (2026-04-15)
- 新增：XX 功能
- 修复：XX 问题
- 优化：XX 流程

### v1.1.0 (2026-04-01)
- 新增：XX 功能

### v1.0.0 (2026-03-15)
- 初始版本
```
