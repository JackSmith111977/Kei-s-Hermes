# 5 种设计模式详解
> 基于 Google Design Patterns 的 Skill 设计模式

**版本：** 1.0.0 | **来源：** Google Design Patterns + BMAD methodology

---

## 模式总览

| 模式 | 核心思想 | 复杂度 | 适用 Skill 类型 |
|------|----------|--------|----------------|
| Tool Wrapper | 封装工具最佳实践 | ⭐ | 库API参考、团队规范 |
| Generator | 从规格生成输出 | ⭐⭐ | 代码生成、文档生成 |
| Reviewer | 对照标准检查评分 | ⭐⭐ | 代码审查、产品验证 |
| Inversion | 从结果反推步骤 | ⭐⭐⭐ | 调研整理、问题诊断 |
| Pipeline | 多步骤顺序执行+追踪 | ⭐⭐⭐ | 部署流程、产品验证 |

---

## 模式 1：Tool Wrapper（工具封装）

### 核心思想
将工具/库/API 的最佳实践封装为可复用的 Skill 指令，让 Claude 始终以正确方式使用工具。

### 适用场景
- 内部库/CLI/SDK 使用指南
- 团队编码规范
- 工具配置标准化

### 结构要素

```yaml
# Tool Wrapper 模式要素
name: tool-wrapper-example
description: 封装 [工具名] 的最佳实践

# 核心章节
- 快速入门（安装 + 第一个示例）
- 常用 API/命令 表格
- 踩坑清单（Gotchas）— 核心！
- 边界情况处理
- 与其他工具配合
```

### 踩坑清单格式

每个陷阱包含三部分：
1. **错误示例** — 展示错误用法
2. **正确示例** — 展示正确用法
3. **原因解释** — 解释为什么会出错

```markdown
### 陷阱 1：[陷阱名称]
❌ **错误用法：**
\`\`\`python
# 错误代码
\`\`\`
✅ **正确用法：**
\`\`\`python
# 正确代码
\`\`\`
**原因：** [解释为什么会出错]
```

### 评估重点
- API 覆盖率（是否覆盖常用功能）
- 示例正确性（代码能否直接运行）
- 陷阱识别度（是否覆盖常见错误）

## 模式 2：Generator（生成器）

### 核心思想
根据输入规格，通过模板填充和规则应用，生成结构化输出（代码/文档/数据）。

### 适用场景
- 从 API spec 生成接口代码
- 从设计稿生成前端代码
- 从变更记录生成 PR 描述
- 从数据生成报表

### 结构要素

```yaml
# Generator 模式要素
name: generator-example
description: 根据 [输入类型] 生成 [输出类型]

# 核心章节
- 输入要求（格式 + 必填字段）
- 生成规则（规则1、规则2...）
- 输出模板（assets/template.md）
- 代码规范/格式规范
- 生成示例
```

### 生成规则编写

每条规则包含：
1. **规则名称** — 简短描述
2. **说明** — 详细解释
3. **示例** — 输入→输出

```markdown
### 规则 1：[规则名称]
**说明：** [描述生成规则]
**示例：**
\`\`\`
输入：[示例输入]
→ 输出：[示例输出]
\`\`\`
```

### 模板变量语法

```
{{变量名}}     — 必填变量
{{变量名|默认值}} — 带默认值的变量
{{{多行变量}}}  — 多行内容变量
```

### 评估重点
- 生成输出正确性
- 模板覆盖度
- 规范遵循度

## 模式 3：Reviewer（审查器）

### 核心思想
对照预定义标准，对输入进行系统性检查，输出评分报告。

### 适用场景
- 代码安全审查
- 代码质量分析
- 产品功能验证
- 文档质量检查

### 结构要素

```yaml
# Reviewer 模式要素
name: reviewer-example
description: 审查 [目标] 的 [方面]

# 核心章节
- 审查范围（类型 + 语言）
- 检查清单（安全/性能/质量）
- 审查流程（读取→检查→报告）
- 报告格式（高危/中危/建议）
- 工具集成（静态分析工具）
```

### 检查清单设计

使用三级严重度：

| 级别 | 符号 | 含义 | 处理方式 |
|------|------|------|----------|
| 高危 | 🔴 | 必须修复 | 阻止发布 |
| 中危 | 🟡 | 应该修复 | 警告但不阻止 |
| 低危 | 🟢 | 建议改进 | 可选修复 |

### 报告格式

```markdown
## 审查结果
### 🔴 高危问题（X 个）
1. [问题描述]
   - 位置：`文件：行号`
   - 风险：[风险说明]
   - 修复：[修复建议]

### 🟡 中危问题（X 个）
...

### 🟢 建议改进（X 个）
...
```

### 评估重点
- 问题发现率（召回率）
- 修复建议质量
- 误报率

## 模式 4：Inversion（逆向工作流）

### 核心思想
从期望结果反推执行步骤，先定义"什么是好的输出"，再推导"如何达到"。

### 适用场景
- 技术调研（先定义调研标准，再收集信息）
- 问题诊断（先定义正常状态，再排查异常）
- 方案设计（先定义目标，再规划路径）

### 结构要素

```yaml
# Inversion 模式要素
name: inversion-example
description: 通过逆向工作流 [达成目标]

# 核心章节
- 期望结果定义（什么是成功的输出）
- 评估标准（如何判断结果好坏）
- 信息收集策略（按优先级收集）
- 分析方法（对比/归纳/演绎）
- 输出模板
```

### 逆向工作流步骤

```
Step 1: 定义期望结果
  → "好的输出应该包含 X、Y、Z"

Step 2: 制定评估标准
  → "用 A、B、C 来判断是否达到期望"

Step 3: 收集信息
  → 按来源优先级收集（官方文档 > 技术博客 > 社区）

Step 4: 分析对比
  → 对比不同来源，识别共识和分歧

Step 5: 生成报告
  → 按输出模板组织内容
```

### 评估重点
- 结果定义清晰度
- 信息覆盖完整度
- 分析逻辑严密性

## 模式 5：Pipeline（流水线）

### 核心思想
将复杂任务分解为多个顺序执行的步骤，每步有明确的输入/输出和检查点。

### 适用场景
- 部署流程（备份→部署→验证→监控）
- 产品验证（准备→自动化测试→手动验证→报告）
- 代码生成（解析→规划→生成→验证→输出）

### 结构要素

```yaml
# Pipeline 模式要素
name: pipeline-example
description: 通过 Pipeline 模式 [完成任务]

# 核心章节
- 硬性门控（Gating Instructions）
- 进度追踪架构（progress.yaml）
- 步骤流程（Step 1 → Step 2 → ...）
- 通过标准
- 异常处理（HALT/回滚）
```

### 硬性门 Gate 规则

```
按顺序执行每一步。如果某一步失败，不得继续。

DON'T:
- 🛑 不要跳过任何步骤
- 🛑 不要合并多个步骤
- 🛑 不要在未通过检查点时继续
- 🛑 不要加载多个 step 文件同时执行
- 🛑 不要在未验证前序步骤完成时开始新步骤
```

### 检查点类型

| 类型 | 触发条件 | 执行方式 |
|------|----------|----------|
| `user-confirm` | 需要用户确认 | 停止执行，等待用户输入 |
| `self-check` | Agent 自检 | 执行自检逻辑，通过后自动继续 |
| `auto-verify` | 脚本/工具验证 | 运行验证脚本，通过后自动继续 |

### 进度追踪文件（progress.yaml）

```yaml
stepsCompleted:
  - step-01-init
  - step-02-prepare
currentStep: step-03-execute
nextStep: step-04-verify
checkpoint: auto-verify
status: in-progress  # in-progress | completed | failed | halted
createdAt: 2026-04-27T10:00:00
updatedAt: 2026-04-27T10:30:00
```

### 评估重点
- 步骤完整性
- 门控有效性
- 异常处理覆盖度

## 模式组合

### 常见组合

| 组合 | 适用场景 | 说明 |
|------|----------|------|
| Tool Wrapper + Reviewer | 代码审查 | 封装检查工具 + 评分报告 |
| Generator + Pipeline | 代码生成 | 模板生成 + 多步验证 |
| Generator + Inversion | 文档生成 | 逆向定义标准 + 正向生成 |
| Pipeline + Reviewer | 部署流程 | 多步执行 + 质量检查 |
| Inversion + Generator | 调研整理 | 逆向定义标准 + 正向收集 |

### 组合设计原则

1. **单一职责** — 每个模式解决一个问题
2. **顺序清晰** — 模式之间有明确的执行顺序
3. **状态共享** — 通过 progress.yaml 共享状态
4. **失败隔离** — 一个模式失败不影响其他模式的定义

## 选择指南

```
需要封装工具使用方式？
└─ 是 → Tool Wrapper

需要根据输入生成输出？
└─ 是 → Generator
    └─ 需要多步验证？ → + Pipeline

需要对照标准检查？
└─ 是 → Reviewer

需要从结果反推步骤？
└─ 是 → Inversion

需要严格顺序执行？
└─ 是 → Pipeline
    └─ 需要质量检查？ → + Reviewer
```
