# deep-research — 深度调研工作流 v2.0

> **核心理念**：调研不是"把学到的东西写下来"，而是**面向受众的结构化交付**。
> 每次调研的最终产出必须是**可阅读的报告（PDF）**或**可展示的演示文稿（PPT）**，而不是零散的笔记。
>
> **核心方法论**：BLUF（结论先行）+ MECE（完全穷尽、互不重叠）+ 金字塔原理（3层支撑）
>
> **v2.0 重大更新**：实战验证。增加 PDF/PPT 自动生成流程、子代理并行产出策略、价格数据交叉验证模板。

---

## 〇、调研流程总览

```
[Phase 0] 需求理解 — SCQA 定义范围
    ↓
[Phase 1] 信息收集 — 多轮联网搜索
    ↓
[Phase 2] 过滤交叉验证 — 来源分级 + 价格验证
    ↓
[Phase 3] 结构化分析 — MECE 分类 + 对比矩阵 + 路线图
    ↓
[Phase 4] 报告产出 — 自动生成 PDF + PPT
    ↓
[Phase 5] 交付汇报 — 向主人展示产出路径
```

## Phase 0: 需求理解

**在任何搜索之前，必须先回答以下问题：**

| 问题 | 作用 |
|------|------|
| 👤 受众是谁？ | 决定报告风格（技术深 vs 商务浅） |
| ❓ 回答什么问题？ | 一句话定义调研目标 |
| 📊 需要什么产出？ | 对话汇报 / PDF 报告 / PPT 演示 |
| ⏱ 时间要求？ | 决定调研深度和搜索轮次 |

**SCQA 框架（用于定义调研范围）：**

| 元素 | 问题 | 示例 |
|------|------|------|
| **S**ituation 现状 | 当前是什么情况？ | Tavily 是 Hermes 唯一的搜索 Provider |
| **C**omplication 挑战 | 有什么问题？ | Tavily 额度有限，用完就没了 |
| **Q**uestion 问题 | 要解决什么？ | Tavily 用完后的替代方案是什么？能不能造本地轮子？ |
| **A**nswer 答案 | 结论是什么？ | Brave Search MCP（3 分钟配置），SearXNG 自托管（长期方案） |

## Phase 1: 信息收集

### 搜索策略

1. **复杂主题** → 拆分为 3-5 个原子子主题，每个独立搜索
2. **交叉源对比** → 从 3 个角度搜集：官方文档/社区/开源
3. **搜索技巧**：
   - 英文搜索优先（技术资料更多）
   - 中英文混合（实战经验中文更好）
   - 搜索关键词用 `2025` / `2026` 过滤时效

### 搜索轮次 — 最小轮次 + 递进关系

**铁律**：调研必须**至少完成 2 轮搜索**（广度+深度）。第 3 轮按需执行。

| 轮次 | 递进等级 | 最少搜索数 | 目的 | 是否必做 |
|:---:|:---|:---:|:---|:---:|
| 第 1 轮 | 🌐 **广度扫描** | 3-5 个搜索 | 全覆盖，获取全景，识别所有候选方案 | ✅ **必做** |
| 第 2 轮 | 🔍 **深度挖掘** | 2-3 个搜索 | 深入细节，交叉验证价格/核心数据 | ✅ **必做** |
| 第 3 轮 | ✨ **精炼验证** | 1-2 个搜索 | 验证异议，查漏补缺，补充遗漏角度 | ⭕ 按需 |

**递进关系**：
- 第 1 轮：覆盖全景，不遗漏任何候选 → 产出一个**候选清单**
- 第 2 轮：对第 1 轮的候选清单逐项深挖 → 验证价格、架构、优劣
- 第 3 轮：检查是否有遗漏的角度 → 补充第 2 轮中发现的信息缺口

**强检查**：
- ❌ 禁止只做 1 轮搜索就下结论
- ❌ 禁止第 2 轮搜索时又重新开始（必须基于第 1 轮的发现深入）
- ✅ 每个轮次的搜索结果标注轮次编号（Round 1/2/3）方便追溯

### 必须保存原始素材

每次搜索后保存原始结果到 `~/.hermes/learning/raw_search_results_{topic}.md`，包含：
- 来源 URL
- 发布日期
- 核心要点（3-5 条）

### 🚩 实战教训（v2.0 新增）

第一轮搜索往往不够——**价格信息经常变化**（如 Brave 2026年2月取消免费层），必须做第二轮搜索专门验证价格数据。否则报告中的价格可能已经过时。

## Phase 2: 过滤与交叉验证

### 来源分级

| 等级 | 来源类型 | 示例 | 信任度 |
|------|---------|------|--------|
| 🥇 | 官方文档 / 官方 API 文档 | GitHub README、API Docs | ✅ 最高 |
| 🥇 | 权威基准测试 | AIMultiple、Proxyway | ✅ 高 |
| 🥈 | 技术博客 / 社区讨论 | Medium、Reddit、Dev.to | ⚠️ 需交叉验证 |
| 🥉 | AI 聚合 / 个人博客 | 无原文链接的内容 | ❌ 仅作线索 |

### 🔗 铁律：每项事实必须携带可验证链接

**这是报告质量的核心门禁。每项事实、数据、结论都必须在报告中标注可点击的源链接。**

#### 1. 链接标注规范

| 事实类型 | 标注要求 | 示例 |
|:---------|:---------|:------|
| **报价/定价** | 官方定价页面 URL + 最后验证日期 | `[OpenAI Pricing](https://openai.com/pricing) — 2026-05确认` |
| **技术架构** | 官方文档或论文 URL | `[AIOS arxiv 2403.16971](https://arxiv.org/abs/2403.16971)` |
| **统计数据** | 来源 URL + 发布时间 | `[Colaberry AI — 16,896 skills](https://colaberry.ai/aixcelerator/skills/ontology) — 2026年` |
| **对比结论** | 至少 2 个独立来源交叉验证 | `[Source A](url1) + [Source B](url2) — 交叉验证一致` |
| **业界实践** | 产品/项目主页或文档 | `[MoFA Microkernel](https://mofa.ai/mofa/concepts/microkernel.html)` |

#### 2. 格式标准（HTML 报告）

```html
<!-- 内联引用 — 正文中直接链接 -->
<p>Colaberry AI 在生产中管理 <a href="https://colaberry.ai/aixcelerator/skills/ontology">16,896 个 skill</a>，
其三层设计与我们的架构一致。</p>

<!-- 脚注引用 — 用 sup 标记 -->
<p>微内核架构在 Agent 系统中已被验证有效<sup><a href="https://arxiv.org/abs/2512.01610">[1]</a></sup>。</p>

<!-- 对比矩阵中的链接 -->
<tr>
  <td>方案 A</td>
  <td>$0.01/次 <a href="https://example.com/pricing">[定价页]</a></td>
</tr>
```

#### 3. 格式标准（Markdown 报告）

```markdown
Colaberry AI 在生产中管理 [16,896 个 skill](https://colaberry.ai/aixcelerator/skills/ontology)，
其三层设计与我们的架构一致。

微内核架构在 Agent 系统中已被验证有效[^1]。

[^1]: Agent-Kernel, arxiv 2512.01610, https://arxiv.org/abs/2512.01610
```

#### 4. 参考文献章节

报告末尾必须包含**完整参考文献**，按分类分组的链接清单：

```markdown
## 参考文献

### 学术论文
- [1] AIOS: LLM Agent Operating System, arxiv 2403.16971
- [2] Agent-Kernel: MicroKernel MAS, arxiv 2512.01610
- ...

### 业界实践
- Colaberry AI Skill Ontology: https://colaberry.ai/aixcelerator/skills/ontology
- OntoSkills: https://ontoskills.sh/docs/overview/
- ...
```

#### 5. 质量门禁检查

报告生成后必须自检：

- [ ] 每个事实/数据点都有 URL 来源？
- [ ] 所有 URL 可点击（HTML 的 `<a href>` 或 Markdown 的 `[]()`）？
- [ ] 定价信息标注了"最后验证日期"？
- [ ] 参考文献章节存在且完整？
- [ ] 结论性陈述至少 2 个独立来源交叉验证？

### 交叉验证标准

- **核心结论**必须找到 ≥2 个独立来源佐证
- **价格数据**必须来自官方价格页面
- **时效性**：技术内容超过 12 个月 → 标记为"可能过时"

### 🚩 价格数据交叉验证模板（v2.0 新增）

价格信息是调研中最容易出问题的环节。每次收集价格后必须：

1. 找到该 API 的**官方定价页面 URL**
2. 搜索"取消免费层""涨价"等新闻确认最近变动
3. 对比至少 2 个第三方汇总来源（如 CostBench、Prospeo）
4. 记录最后验证日期

## Phase 3: 结构化分析

### MECE 分类法

将所有方案/选项按 MECE 原则分类：

```
所有方案
├── 方案类 A（特征 A）
│   ├── 具体方案 1
│   └── 具体方案 2
├── 方案类 B（特征 B）
│   ├── 具体方案 3
│   └── 具体方案 4
└── 方案类 C（特征 C）
    └── 其他（MECE 兜底）
```

### 多维度对比矩阵

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 免费额度 | 描述 | 描述 | 描述 |
| 月费(少量) | $ | $ | $ |
| 月费(大量) | $ | $ | $ |
| 部署复杂度 | ✅/⚠️/❌ | ✅/⚠️/❌ | ✅/⚠️/❌ |
| 搜索质量 | ⭐/⭐⭐/⭐⭐⭐ | ⭐/⭐⭐/⭐⭐⭐ | ⭐/⭐⭐/⭐⭐⭐ |
| 核心特性 | 一句话 | 一句话 | 一句话 |

**配色约定**：
- ✅ 绿色 = 优势
- ⚠️ 黄色 = 中性
- ❌ 红色 = 劣势

### 价格按查询量级对比表（v2.0 新增）

这是调研中**最核心**的对比维度。必须按使用量分级展示：

| 月查询量 | 方案 A | 方案 B | 方案 C |
|---------|--------|--------|--------|
| 1,000 | $ | $ | $ |
| 10,000 | $ | $ | $ |
| 100,000 | $ | $ | $ |
| 500,000 | $ | $ | $ |

**为什么要按量级展示？** 不同方案在不同量级的性价比曲线完全不同。如 API 方案是线性增长，自托管是固定成本。

### 建议路线图

按时间线划分建议：

| 时间 | 做什么 | 成本 | 收益 |
|------|--------|------|------|
| 🚀 立即（今天） | 最低成本方案 | 最低 | 快速解决 |
| 🎯 短期（1-2 周） | 适度投入方案 | 中等 | 提升质量 |
| 🌟 长期（按需） | 最优方案 | 最高 | 根本性解决 |

## Phase 4: 报告产出

### 产出选择决策

```
受众是谁？
├─ 给自己/技术团队 → PDF 详细报告
│     └─ 格式：结构化报告（10-12 页）
├─ 给老板/客户 → PPT 演示
│     └─ 格式：精简 PPT（7 页左右）
├─ 系统分析/产品规划 → Epic 文档
│     └─ 格式：Epic + 用户故事（.md 文件）
│         适用场景：技术方案分析、架构决策、产品路线规划
│         包含：概述 → 用户故事(含验收标准) → 架构变更
│             → 关键设计决策 → 实施路线图 → 风险评估
└─ 两者都要 → 先出 PDF，再摘要成 PPT
```

### 报告标准结构（PDF 版 → WeasyPrint）

| 章节 | 内容 | 约占比 |
|------|------|--------|
| 1. 封面 | 标题、副标题、日期、作者 | 1 页 |
| 2. 目录 | 自动生成（HTML 手工目录） | 1 页 |
| 3. 执行摘要 | BLUF 先行 + 3 个核心发现 + 建议 | 1 页 |
| 4. 背景与范围 | 为什么做、怎么做的 | 1-2 页 |
| 5. 方法论 | 搜索策略、评估标准、局限 | 1 页 |
| 6. 方案全景 | MECE 分类展示 | 1-2 页 |
| 7. 方案对比 | 多维度对比矩阵 | 2-3 页 |
| 8. 价格分析 | 按查询量级对比 | 1-2 页 |
| 9. 集成方案 | 架构图 + 配置代码（如适用） | 1-2 页 |
| 10. 推荐路线 | 立即/短期/长期 | 1 页 |
| 11. 结论与行动 | 一句话总结+下一步 | 1 页 |
| 12. 附录 | 参考文献、原始数据 | 1-2 页 |

**生成 PDF 的步骤**：
1. 用 f-string 或 Jinja2 生成完整 HTML 模板（含 CSS 样式）
2. **关键**：所有数据必须在 HTML 字符串中硬编码，不要动态查询
3. 使用 `/usr/bin/python3`（系统 Python，不是 venv）
4. 调用 `from weasyprint import HTML; HTML(string=...).write_pdf(path)`
5. 验证 PDF 文件大小（正常范围：500KB-2MB）

### 🚩 PDF 生成实战经验（v2.0 新增）

1. **WeasyPrint 必须用系统 apt 版本**，pip 版本可能缺系统依赖
2. 检查方式：`which weasyprint` 存在 + `/usr/bin/python3 -c "from weasyprint import HTML; print('OK')"`
3. **中文必须用 `@font-face` 指定字体**，否则 WeasyPrint 默认字体不显示中文
4. 字体推荐：`"WenQuanYi Micro Hei"` 或 `"Noto Sans CJK SC"`（系统已安装）
5. PDF 生成脚本保存到 `~/.hermes/research/output/generate_report.py`，方便复现
6. **子代理不适合生成长 HTML 模板**——容易超时。PDF 生成应在主会话中由 boku 直接完成

### 演示标准结构（PPT 版 → python-pptx）

| 页码 | 内容 | 设计要点 |
|------|------|----------|
| 1 | **封面** | 深蓝背景+橙色装饰线+大标题 |
| 2 | **执行摘要 = 结论先行** | 黄色结论框 + 3 张发现卡片 |
| 3 | **背景挑战** | 列表式展示痛点 |
| 4 | **MECE 方案全景** | 三栏卡片分类 |
| 5 | **方案对比矩阵** | 完整表格（8 维度 × 5 方案） |
| 6 | **推荐路线** | 四张时间线卡片 |
| 7 | **总结与行动** | 深蓝背景+白色清单 |

**生成 PPT 的步骤**：
1. 用 python-pptx 创建
2. 步骤详见 `pptx-guide` skill

### 🚩 PPT 生成实战经验（v2.0 新增）

1. `python-pptx` 安装在系统 Python 3.12 中（`/usr/bin/python3`），venv 中没有
2. 表头用深蓝色背景+白色文字，数据行交替浅灰色
3. PPT 文件通常很小（30-50KB），因为 python-pptx 只存结构化数据不存图片
4. PPT 脚本也保存到 `~/.hermes/research/output/generate_ppt.py`

### 🚩 调研交付物不等于实施指令（v2.1 新增）

调研完成后，**不要默认主人会同意你的方案**或直接跳到实施。正确的交接流程是：

```
调研完成 → 回答核心问题 → 验证可行性 → 向主人汇报
    → 主人判断（"对的"/"方向可以"/"实施"）
    → 只有主人明确说实施再动手
```

**参考文件**: `references/html-retroactive-source-linking.md` — 已有 HTML 报告追溯添加可点击链接的操作手法

**常见错误：**
- ❌ 调研完直接说"我来实施" → 应该先问"主人觉得这个方向对不对？"
- ❌ 调研和方案混在一起 → 应该清晰区分"调研发现"和"建议方案"
- ❌ 默认主人接受你的设计假设 → 应该先确认项目的核心约束和需求前提
- ❌ 自己决定实施路径 → 要在汇报中给选项，让主人做决策

**正确做法：**
- 调研产出中包含明确的**可行性结论**（✅ 可行 / ⚠️ 有风险 / ❌ 不可行）
- 列出**需要主人判断的决策点**
- 用问句结尾："主人觉得这个方向对吗？要不要进入实施？"

### 🚩 子代理并行产出的正确策略（v2.0 新增）

**不要用子代理生成长代码脚本**——子代理容易超时或上下文不足。正确的策略是：

1. **boku 主会话**编写完整的 HTML/Python 模板
2. 将模板写入 `.py` 文件
3. 用 `terminal` 直接执行文件
4. 同理，PPT 也是 boku 主会话写代码、terminal 执行

## 五、AI Agent 调研铁律

### Do ✅
- ✅ 调研**之前**先问主人：受众是谁？需要什么格式？
- ✅ 结论先行 — 第一句话就给出核心答案
- ✅ MECE 分类 — 方案之间不重叠、全覆盖
- ✅ 对比矩阵 — 至少 3 个维度
- ✅ 价格按查询量级对比 — 1K/10K/100K/500K 四个量级
- ✅ 推荐意见 — 不只是罗列，要有判断
- ✅ 行动路线 — 立即/短期/长期
- ✅ 交叉验证价格 — 所有价格 ≥2 个来源

### Don't ❌
- ❌ 调研结果直接扔给主人（必须出报告！）
- ❌ 所有结论混在一起（必须结构化！）
- ❌ 不列来源（每个数据点都要注明出处）
- ❌ 不交叉验证价格（价格经常变化，要确认最新数据）
- ❌ 只说问题不说建议（必须给出推荐意见）
- ❌ 第一轮搜索后就停止（必须做第二轮验证价格）
